## 2. 目标与非目标

### 2.1 目标

- 用同一套领域、API 和 UI 组件支持美妆、服装两个独立商城的商品内容运营。
- 交付品牌、类目树、属性模板、SPU、SKU、媒体、三语翻译、合规资料、商品版本和首页装修的数据基础。
- 所有商城业务表包含不可为空的 `store_id`，用复合外键、复合唯一约束、RLS 和服务端 `StoreContext` 阻断跨商城关联与访问。
- 越南语内容和适用合规资料不完整时禁止上架；普通运营不能绕过合规审核。
- 交付后台商品内容管理，以及买家端首页、分类、品牌馆、商品列表和商品详情的真实数据库浏览链路。
- 为 M3 库存/搜索/计价和 M4 订单快照提供稳定 ID、整数 VND 价格字段与可追溯商品版本来源。

### 2.2 非目标

- 不实现仓库、库存余额、库存锁、购物车、促销、优惠券、搜索索引或服务端结算；这些属于 M3。
- 不实现地址、订单、COD、支付、物流、售后、分享调用和报表；这些属于 M4-M7。
- 不接入生产对象存储或图片处理供应商；M2 只定义稳定媒体端口和可验证的 local/test S3 兼容实现。
- 不用静态假商品或永远成功上传冒充真实链路。
- 不实现第三方商家、店铺入驻、商家结算或跨商城共享商品记录。
- 不在字段和前端代码中写死“护肤 - 化妆品”是否应为“化妆水”；类目由商城配置和种子维护。

## 3. 领域边界与不变量

### 3.1 聚合边界

- 品牌：商城内独立，包含三语内容、媒体、启用/停用和排序信息。
- 类目：商城内树结构，支持主父级、排序、启用/停用、三语内容、媒体和属性模板绑定。
- 属性模板：按商城和行业管理属性定义、可选值、单位、筛选/规格用途与必填规则。
- 商品：SPU 是发布、翻译、合规和版本聚合根；SKU 是实际销售单位，但 M2 不承载库存。
- 媒体：保存商城归属、对象键、类型、尺寸、替代文本和处理状态；业务对象通过关联表引用。
- 装修：页面与模块按商城、页面、语言、时间窗口和排序管理，跳转目标使用受校验的内部目标类型。

### 3.2 必须由数据库保护的不变量

- 品牌编码、类目编码、商品编码和 SKU 编码只在各自商城内唯一。
- 商品主品牌、主类目、辅助类目、属性模板、SKU、媒体和合规记录必须与商品属于同一商城。
- 类目父子关系必须同商城；服务层额外防止自引用、循环和超过批准的最大深度。
- SKU 金额使用非负整数 VND；不得使用浮点类型。商品展示价格范围由启用 SKU 计算，不作为可信结算快照。
- 一个 SKU 的规格组合在同一商品内唯一；规格值必须来自该商品绑定模板允许的规格属性。
- 已被商品引用的品牌/类目只能停用；商品、品牌、类目默认软删除/停用，不物理破坏历史引用。
- 媒体对象键必须包含环境和 `store_id`，引用前再次校验商城归属。
- 发布版本不可变；编辑草稿不能静默修改已发布版本。订单后续只引用不可变版本或保存完整快照。

### 3.3 上架门禁

上架命令在同一数据库事务中重新读取完整聚合并返回可解释的门禁结果，至少校验：

- 商品、主品牌、主类目和至少一个 SKU 均启用。
- 越南语名称、卖点/详情和所有发布必填内容存在；中文、英文可回退越南语，但记录完整度。
- 主图、SKU 销售价、规格组合及要求的行业属性完整。
- 美妆、美瞳、口腔和私密护理命中的合规规则均有有效、已审核且未过期的资料。
- 合规审核者与提交者按权限分离；普通 `store.catalog.manage` 不能授予审核通过。
- 上架生成不可变发布版本和审计事件；门禁失败不部分更新状态。

## 4. 数据模型设计

实际字段字典须在 M2.1 固化；计划模型如下。所有商城表均带 `store_id`，时间使用带时区时间戳，稳定内部主键使用 UUID。

| 模型                           | 关键字段与约束                                                                                     |
| ------------------------------ | -------------------------------------------------------------------------------------------------- |
| `brands`                       | `id`、`store_id`、`code`、状态、国家/地区、官网、推荐、排序、软删除；唯一 `(store_id, code)`       |
| `brand_localizations`          | `store_id`、`brand_id`、`locale`、名称、介绍、分享文案；主键 `(store_id, brand_id, locale)`        |
| `categories`                   | `id`、`store_id`、`parent_id`、`code`、层级、排序、状态、软删除；同商城复合父键                    |
| `category_localizations`       | 三语名称、描述、分享文案；复合主键包含商城与语言                                                   |
| `attribute_templates`          | 商城、行业、编码、当前版本和状态的稳定根                                                           |
| `attribute_template_versions`  | 模板不可变版本、名称、激活状态和审核字段                                                           |
| `attribute_definitions`        | 模板版本、编码、数据类型、用途（描述/筛选/规格）、必填、多选、单位、排序、校验规则 JSON            |
| `attribute_options`            | 属性定义、稳定编码、三语标签、排序、启用状态                                                       |
| `category_attribute_templates` | 类目与模板版本的同商城关联                                                                         |
| `products`                     | 商城、商品编码、品牌、主类目、状态、推荐、COD/售后策略引用、当前草稿/发布版本、软删除              |
| `product_secondary_categories` | 商品与辅助类目的同商城关联                                                                         |
| `product_localizations`        | 三语名称、副标题、卖点、详情、使用/洗护说明、SEO/分享内容                                          |
| `product_attribute_values`     | 商品、属性定义及类型化值；服务层和约束验证类型/选项归属                                            |
| `skus`                         | 商城、商品、SKU 编码、条码、整数 VND 市场/销售/成本价、重量、尺寸、状态；唯一 `(store_id, code)`   |
| `sku_option_values`            | SKU 与规格选项关联；规格组合另存规范化 hash 并在商品内唯一                                         |
| `media_assets`                 | 商城、对象键、MIME、字节数、宽高、checksum、处理状态、替代文本、软删除                             |
| 专用媒体关联表                 | `brand_media`、`category_media`、`product_media`、`sku_media`、`page_module_media`，均使用复合外键 |
| `compliance_requirements`      | 商城、行业/类目、资料类型、必填条件、有效期和上架阻断规则，版本化配置                              |
| `compliance_records`           | 商品/批次、资料类型、编号、签发/到期、媒体、提交/审核人、审核状态与意见                            |
| `product_versions`             | 商品、递增版本、不可变聚合 JSON、内容 hash、创建人、发布状态和发布时间                             |
| `pages`                        | 商城、页面编码、状态、当前发布版本和乐观锁版本；唯一 `(store_id, code)`                            |
| `page_versions`                | 页面不可变草稿/发布版本                                                                            |
| `page_modules`                 | 商城、页面版本、模块类型、排序、时间窗口、状态、主题配置和受校验跳转目标                           |
| `page_module_localizations`    | 模块三语标题、摘要、按钮文案及内容配置                                                             |

JSON 只用于版本快照、展示配置和可演进校验规则；品牌、类目、商品、SKU、金额、状态和关系等核心可查询字段不塞入不受约束的 JSON。

## 5. API 与权限变化

### 5.1 权限目录

新增商城范围权限，继续默认拒绝：

| 权限                      | 用途                                     |
| ------------------------- | ---------------------------------------- |
| `store.catalog.read`      | 后台读取品牌、类目、属性模板、商品和 SKU |
| `store.catalog.manage`    | 创建和编辑草稿、停用实体、管理媒体关联   |
| `store.catalog.publish`   | 提交审核、上架、下架和定时发布           |
| `store.compliance.read`   | 读取合规要求和记录                       |
| `store.compliance.review` | 审核/驳回合规资料，不隐含普通编辑权      |
| `store.content.read`      | 读取页面和模块配置                       |
| `store.content.manage`    | 编辑、排序、定时和发布装修内容           |

种子只把新权限授予可识别的 local/test 系统商城角色；不修改真实管理员授权。所有写操作写审计日志，批量操作记录请求条件、成功/失败对象和前后版本。

### 5.2 买家 API

- `GET /v1/catalog/home`
- `GET /v1/catalog/brands`、`GET /v1/catalog/brands/{brand_code}`
- `GET /v1/catalog/categories`、`GET /v1/catalog/categories/{category_code}`
- `GET /v1/catalog/products`、`GET /v1/catalog/products/{product_code}`

买家请求从受信 `X-Store-Code` 解析商城上下文，只返回已发布、当前时间有效且商城一致的数据。语言由受支持的请求语言解析，缺失内容回退越南语并返回实际使用语言。列表使用游标分页和白名单排序；M2 只提供结构化筛选，不宣称实现 M3 全文搜索。

### 5.3 管理 API

- `/v1/admin/catalog/brands`
- `/v1/admin/catalog/categories`
- `/v1/admin/catalog/attribute-templates`
- `/v1/admin/catalog/products` 与 `/v1/admin/catalog/products/{id}/skus`
- `/v1/admin/catalog/products/{id}/submit|publish|unpublish`
- `/v1/admin/compliance/requirements` 与 `/v1/admin/compliance/records/{id}/review`
- `/v1/admin/media/uploads`、`/v1/admin/media/{id}`
- `/v1/admin/content/pages` 与 `/v1/admin/content/pages/{id}/modules`

所有管理 API 目标商城由已验证令牌、商城代码与路径/查询 `store_id` 三方一致性确定；不信任请求体内的 `store_id`。创建/更新 DTO 使用 Zod，响应和错误码同步到 OpenAPI。

媒体上传采用两步协议：服务端校验类型、大小和商城后签发短期上传目标；客户端上传后以 checksum/元数据完成确认。local/test 连接现有 MinIO，生产对象存储仅保留配置端口，不写假成功实现。

## 8. 迁移、兼容与回滚

- M2 采用新增表/列/索引的扩展迁移，不破坏 M1 接口和数据。
- 新枚举优先使用数据库检查约束或只追加值；不得依赖无法安全回滚的枚举改名。
- 大索引使用适合部署窗口的并发/分步方案；迁移不得在事务中进行外部对象存储调用。
- API `/v1` 只做向后兼容新增；M1 客户端在 M2 部署后继续可用。
- 渠道/页面功能使用商城配置关闭；应用可回滚到兼容 M1 的版本，但数据库含业务数据后保留新增 schema。
- local/test 的 `down.sql` 只用于无真实 M2 数据环境，并先验证绝对数据库目标；staging/production 采用向前修复。
- 已上传但未被确认引用的对象由带商城维度的清理任务处理；删除业务记录不直接删除仍有引用的对象。

## 9. 测试与验收

### 9.1 自动化

- 单元：状态转换、类目循环、规格组合 hash、三语回退、上架门禁、整数 VND 和媒体校验。
- 数据库集成：每张商城表 RLS、复合外键/唯一约束、不可变版本、软删除和并发乐观锁。
- API：认证、RBAC、跨商城 ID/代码/媒体篡改、输入边界、分页、批量部分失败和错误响应。
- 安全：上传 MIME 欺骗、超限文件、对象键穿越、未授权签名 URL、合规审核绕过和敏感资料日志脱敏。
- UI/E2E：两套主题、三语、后台 CRUD/审核/装修、买家浏览/SKU 选择、加载/空/错误/无权/冲突/图片失败状态。

- 至少覆盖 360x800、390x844、430x932 和桌面 1440x900。

- 真实 Browser 不可用时只能报告构建和源码检查，不能把 UI/E2E 标为通过。
