## 2. 目标与非目标

### 2.1 目标

- 建立商城隔离、仓库隔离且可审计的库存事实来源，明确现有量、锁定量与可售量，并提供防超卖、幂等预留/释放/确认原语。
- 使用 PostgreSQL FTS、`unaccent`、`pg_trgm` 和受控规范化检索文档完成三语搜索、越南语有/无变音符号检索、联想、筛选、排序、历史和热门词。
- 建立版本化促销与优惠券规则，使用整数 VND 的纯价格引擎输出逐行、逐规则、可解释且确定性的商品金额报价。
- 建立每会员、每商城独立的持久购物车，覆盖 SKU/数量/选择状态、下架/失效/库存不足/价格变化提示和服务端重算。
- 交付美妆/服装共用组件的移动端搜索、筛选、SKU 与购物车体验，以及仓库库存和促销管理端工作台。
- 为 M4 下单锁库存和最终结算保存稳定的领域端口与契约，但不伪造尚未存在的订单、运费或核销数据。

### 2.2 非目标

- 不创建地址、结算单、订单、订单快照、COD、支付、物流、退款或售后状态机。
- 加入购物车不锁库存；只有 M4 的下单命令才能调用 M3 的库存预留原语。
- 不把购物车报价当作最终订单金额。M4 必须重新加载商品、库存、促销、优惠券、地址与运费并重新计价。
- 不伪造销量排序。M3 提供相关度、上新、价格升降序；真实销量排序在 M4 产生订单事实后接入，P0 发布前完成。
- 不伪造运费、偏远地区附加费或包邮后的最终应付金额。M3 可保存包邮规则并返回待 M4 运费报价验证的资格结果。
- 不核销优惠券或消耗使用次数；M4 在创建订单的同一可信事务中重新验证并记录核销。
- 不引入 OpenSearch、Elasticsearch、第三方营销平台、ERP/WMS 或多仓智能分配。
- 不关闭 Zalo 宿主真机、生产对象存储/CDN 或越南专业合规复核等既有上线前置条件。

## 3. 冻结的领域不变量

### 3.1 商城、身份与权限

- 所有 M3 业务表均带非空 `store_id`；仓库、SKU、商品、类目、品牌、会员之间使用包含 `store_id` 的复合外键。
- 公共搜索从可信 `X-Store-Code` 解析商城；购物车只接受商城绑定的会员访问令牌，令牌商城与 Header 不一致时拒绝。
- 管理端沿用 M1 的密码 + TOTP、商城授权和逐商城 RLS 事务；请求体不接受 `store_id`。
- 新增商城权限：`store.inventory.read`、`store.inventory.manage`、`store.inventory.adjust`、`store.promotions.read`、`store.promotions.manage`、`store.promotions.publish`。local/test `store-admin` 种子可获得全部 M3 商城权限，生产既有角色不得被迁移静默扩权。
- 仓库编辑、库存调整、促销发布/暂停/终止和优惠券状态变化均写前后快照审计；库存流水本身只追加且不允许 UPDATE/DELETE。

### 3.2 库存

- `on_hand` 表示账面现有量，`reserved` 表示有效锁定量，`available = on_hand - reserved` 在读取时或数据库生成表达式中派生，不作为可被任意写入的第三份事实。
- 数量使用非负整数；数据库检查 `on_hand >= 0`、`reserved >= 0`、`reserved <= on_hand`。任何余额变化和对应流水必须在同一数据库事务中提交。
- 同一库存动作使用商城范围幂等键；重复请求返回首次结果，不产生第二笔余额或流水变化。
- 多 SKU 操作按稳定 SKU/仓库顺序取得行锁，避免死锁；预留使用 `SELECT ... FOR UPDATE` 或等价条件更新，余额不足时整批失败且不部分锁定。
- 预留状态为 `ACTIVE -> RELEASED | CONSUMED | EXPIRED`，终态不可逆。释放、确认和超时任务使用状态条件与唯一动作键，重复回调/重试无第二次变化。
- 购物车只读取可售量，不创建预留。M3 的预留应用服务为 M4 提供内部端口，并通过并发集成测试验证；M4 才把预留绑定真实订单。
- P0 每商城只有一个启用的默认履约仓库参与买家可售量；schema 允许增加仓库，但多仓分配、调拨与路由属于 P1。

### 3.3 金额、促销与优惠券

- SKU `sale_price_vnd` 是基础单价；客户端提交的单价、折扣、小计或应付金额一律不可信。价格服务只接受 SKU 业务编码、数量和可选优惠券码，并在可信商城事务中重新加载事实。
- 所有金额为非负整数 VND；百分比使用整数基点（1/10000），每一步按冻结的向下取整规则计算，折扣总额不得超过适用金额，结果不得为负数。
- 促销使用稳定根与不可变发布版本。草稿可编辑；发布后规则、目标、时间窗、门槛、金额和叠加策略不可原地修改，变更必须创建新版本。
- 计价应用槽分为 `ITEM`、`ORDER`、`COUPON`、`SHIPPING`；`COUPON` 只允许由优惠券引用的规则版本占用，不作为自动促销桶。每行最多一个商品级优惠，每份报价最多一个订单级自动优惠、一个优惠券和一个运费优惠资格。默认互斥；只有双方规则都显式允许时才能跨槽叠加。
- 同一桶出现多个可用规则时选择对买家 VND 优惠最大者；相同优惠按较小 `priority`、较早版本 ID 的稳定顺序决定，保证重放结果一致。
- 计价顺序：基础商品小计 → 商品级优惠 → 订单门槛判断/订单级优惠 → 优惠券 → 商品应付。M3 返回运费优惠资格但不填造运费；M4 扩展为商品应付 + 运费 + 偏远附加费 - 运费优惠后的最终订单应付。
- 价格响应必须包含基础单价、数量、每项规则 code/version、适用基数、折扣、未适用原因、商品小计/优惠/应付、报价时间和确定性 `quote_hash`。M4 必须把接受时的完整分解写入订单快照。
- 新客资格在 M3 当前零订单基线上可判定为零已完成订单；M4 引入真实订单后改由同商城订单事实计算。优惠券在 M3 仅校验资格，不记录核销。

### 3.4 搜索

- 只索引当前商城可展示的已发布商品；下架、停用、删除或发布版本切换在同一业务事务中更新/移除检索文档，不依赖最终一致前端过滤。
- 每个商品、每种语言一份检索文档，包含商品、品牌、主/辅助类目、可筛选属性和业务标签的显示文本、规范化文本及 `tsvector`。
- 越南语查询同时保留原始小写文本与移除变音符号后的规范化文本；显式处理 `đ/Đ`，不修改展示内容。中文/英文走相同安全规范化并以 trigram 补足分词与联想。
- 相关度优先精确/前缀、FTS 排名，再使用 trigram 相似度；所有原生 SQL 参数化，查询长度、筛选数量和游标大小受限。
- 搜索筛选支持品牌、一级/二级类目、价格范围、行业筛选属性、是否有货和是否有促销；排序支持相关度、上新、价格升降序。无结果推荐只使用同商城已发布商品。
- 搜索历史仅对已认证会员按商城保存，可清空并限制最近条数；热门词只保存商城级聚合规范化词与计数，不保存匿名用户标识、Token 或原始敏感请求数据。

### 3.5 购物车

- 每个 `(store_id, member_id)` 最多一个 `ACTIVE` 购物车；商品行唯一 `(store_id, cart_id, sku_id)`，数量限制在冻结契约内，使用版本或幂等的“设置数量”语义避免重复点击累加。
- 购物车保存加入时基础单价，仅用于检测价格变化；每次读取/修改都重新加载商品发布状态、SKU 状态、当前可售量和服务端报价。
- 失效行不物理丢弃，返回明确原因：商品下架/停用、SKU 停用、不足库存、零库存、价格变化或促销变化。用户可删除、换 SKU、调整数量和选择/取消选择。
- 购物车报价仅包含当前选中且仍可购买的行；有阻断问题时不返回虚假“可结算”状态。购物车转换订单和保留历史行属于 M4。
- Mini App 会话继续只在 React 内存中保存，不写 LocalStorage/SessionStorage。Web E2E 只模拟 Zalo 宿主桥接，仍调用真实 test provider 身份交换、真实 API 与数据库；不拦截库存、搜索、价格或购物车业务 API。

## 4. 数据模型与迁移设计范围

M3.1 先冻结 `docs/database/m3-data-dictionary.md`；预计 M3.2 增加以下商城表，最终字段、索引与状态以冻结字典为准：

| 聚合     | 预计表                                                  | 关键约束                                                                      |
| -------- | ------------------------------------------------------- | ----------------------------------------------------------------------------- |
| 仓库     | `warehouses`                                            | `(store_id, code)` 唯一；每商城至多一个启用默认履约仓库；停用不删除历史       |
| 库存     | `inventory_balances`                                    | `(store_id, warehouse_id, sku_id)` 唯一；非负与 `reserved <= on_hand`；版本列 |
| 库存幂等 | `inventory_operations`                                  | `(store_id, operation_key)` 唯一；记录动作类型、请求 hash、结果和操作者       |
| 库存预留 | `inventory_reservations`、`inventory_reservation_items` | 商城复合关系；状态/到期时间；每动作唯一；终态不可逆                           |
| 库存流水 | `inventory_movements`                                   | 只追加；关联余额、预留/操作；保存现有量与锁定量前后值和差量                   |
| 搜索     | `product_search_documents`                              | `(store_id, product_id, locale)` 唯一；GIN FTS、GIN/GiST trigram 与筛选索引   |
| 搜索行为 | `member_search_history`、`search_query_stats`           | 历史绑定同商城会员并可删除；热门词为商城级聚合                                |
| 促销     | `promotions`、`promotion_versions`、`promotion_targets` | 稳定根、不可变发布版本；类型化目标复合外键；版本与时间窗                      |
| 优惠券   | `coupons`、`member_coupons`                             | 商城内 code 唯一；规则版本、领取/有效状态；M3 不创建订单核销记录              |
| 购物车   | `carts`、`cart_items`                                   | 每会员一个活动购物车；同车 SKU 唯一；数量/选择状态/加入价/版本                |

所有表：

- 使用 UUID 内部 ID、`timestamptz`、整数 VND/数量和显式业务 code；唯一键、索引、缓存/任务键均含商城维度。
- 启用并强制 RLS，使用 `app_security.current_store_id()` 默认拒绝；商城归属更新触发器阻止跨店移动。
- 核心关系使用复合外键；发布促销版本、库存流水和预留终态使用数据库触发器保护不可变性。
- `pg_trgm` 与 `unaccent` 使用 PostgreSQL 17 自带可信扩展；迁移前检查目标数据库 `pg_available_extensions` 和迁移角色 `CREATE` 权限。扩展安装在受控 schema/search path 下，不接受用户可写 schema 注入。

已有数据兼容：

- 迁移不为现有 SKU 虚构库存；无余额即不可售。local/test 种子为两个可识别商城创建默认测试仓库和零余额，专项夹具通过受审初始调整写入测试库存与流水。
- 已发布商品回填检索文档；回填可重跑并按商城/商品幂等。应用提供受审重建命令以修复索引，不把 Redis 当作搜索事实来源。
- M2 公开目录保持向后兼容，新增可售与促销字段为可选向后兼容扩展；默认不因零库存隐藏商品，`in_stock=true` 才只返回有货商品。

## 5. 模块、接口与文件变化

- `packages/i18n/src/`、`tests/integration/`、`tests/e2e/`：三语文案、并发/安全/API 和真实浏览器回归。
- `docs/database/m3-data-dictionary.md`、`docs/api/openapi.m3.yaml`、`docs/security/m3-permission-matrix.md`：随 M3.1 冻结并随实现同步。

## 6. 分阶段实施顺序

### M3.1：字段字典、状态机、价格规则与接口契约冻结

- 产出 M3 数据字典、OpenAPI、权限矩阵和库存/价格纯领域测试。
- 冻结状态转换、表关系、数量/金额边界、促销叠加表、搜索规范化样例、API 错误码及 M4 对接端口。
- 不创建迁移或业务 API；文档和测试审查通过后进入 M3.2。

### M3.2：数据库迁移、RLS、初始回填与基础种子

- 实现全部 M3 表、复合外键、唯一/检查约束、RLS、商城不可变与只追加触发器。

- 演练空库升级、M2 已有数据升级、重复 deploy、种子幂等、扩展缺失失败提示、无真实 M3 数据 down 和有数据向前修复。

### M3.3：仓库、库存、预留原语与管理端工作台

- 实现仓库、余额、流水、调整、受限初始库存导入和库存预警读模型。
- 使用行锁、稳定锁顺序、操作键和请求 hash 实现整批预留、释放、确认与过期原语；worker 只扫描数据库事实并幂等执行。
- 管理端提供三语列表、筛选、调整二次确认、逐行导入报告、加载/空/错误/冲突/重试状态；不提供直接编辑余额。

### M3.4：三语搜索、筛选、历史与买家端搜索体验

- 实现发布索引同步、重建命令、FTS/trigram 检索、联想、筛选、游标排序、历史/清空与热门词聚合。
- Mini App 提供搜索页、底部筛选、价格/品牌/类目/行业属性/库存筛选、无结果推荐和三语状态。
- 验证越南语有/无变音符号、`đ`、中文/英文、跨商城哨兵、恶意查询、索引缺失恢复和常见移动视口。

### M3.5：促销、优惠券、价格引擎与管理端工作台

- 实现促销草稿/发布版本、类型化目标、时间窗、互斥/叠加、优惠券资格和发布/暂停/终止审计。
- 先以大量纯单元/性质测试固定整数 VND、舍入、上限、门槛、最优规则和确定性，再开放价格 API。
- 管理端提供三语规则编辑、适用范围、时间窗、预览分解和高风险发布确认；不记录伪造核销。

### M3.6：会员购物车、目录可售/促销投影与移动端 UI

- 实现活动购物车、设置数量、选择、删除、SKU 替换、价格/促销/库存重算和失效原因。
- Mini App 将商品详情主操作接入真实购物车，增加购物车角标、页面和底部导航；未登录时进入 Zalo 身份流程，授权失败保留可恢复提示。

### M3.7：并发、安全、E2E、迁移演练与阶段收口

## 7. 迁移、兼容与回滚

- M3 采用向后兼容新增表/索引/可选响应字段，不删除或重命名 M1/M2 表和接口。
- 搜索回填按商城和商品分批、可重试；大表索引在生产规模评估后使用部署窗口或并发索引方案，迁移不跨外部网络调用。
- 无库存余额的现有 SKU 视为不可售，不自动写“无限库存”或任意初始量。正式初始余额只能通过受审导入/调整产生同事务流水。
- 搜索文档和热门聚合是可重建派生数据，可删除后重建；库存流水、发布促销版本、购物车事实和预留终态不可用回滚脚本删除。
- 仅在确认目标为无真实 M3 数据的 local/test 数据库时允许人工执行 `down.sql`。一旦产生库存流水、促销发布版本或购物车数据，只允许向前修复。
- 应用回滚到 M2 时保留 M3 schema 与数据，停止 M3 worker/API 路由；M2 目录继续工作。库存错误通过引用原动作的反向调整修复，不 UPDATE/DELETE 流水。

## 8. 测试与验收门禁

### 8.1 单元与性质测试

- 库存状态转换、差量守恒、预留/释放/确认/过期幂等和非法终态。
- 所有促销类型、叠加矩阵、互斥、时间边界、门槛、基点舍入、折扣封顶、零值/极值、无效券和确定性 tie-break。
- 搜索规范化覆盖越南语组合/预组合字符、去变音、`đ/Đ`、中文、英文、空白和长度边界。
- 购物车数量、选择、失效、价格变化与可结算阻断规则。

### 8.2 数据库、并发与 API 集成

- 每张表的强制 RLS、复合外键、商城归属不可变、同编码跨店允许和跨店关系拒绝。
- 高并发预留总量不超过现有量；重复调整/预留/释放/确认/过期不产生第二笔流水；批量失败不部分写入。
- 促销发布版本不可变，优惠目标不能跨店，前端金额篡改不影响服务端报价。
- 搜索只返回同商城已发布商品，索引同步/回填/重建一致；历史不能跨会员或跨商城读取。
- 购物车令牌商城绑定、重复设置数量、并发版本冲突、跨店 SKU/行 ID、下架/停用/零库存和促销变化。

### 8.3 UI/E2E

- 管理端：库存读/调整/导入、促销草稿/发布、RBAC 拒绝、冲突与二次确认。
- 买家端：两商城三语搜索/筛选、商品详情可售与促销、真实身份交换、加入购物车、SKU/数量/选择/删除、失效与重算。
- 视口：至少 Pixel 7 Chromium、iPhone 13 WebKit 和管理端 1440×900；无横向溢出，焦点可见，加载/空/错误/重试完整。
- E2E 不拦截 M3 业务 API；只在 Web 预览中模拟 Zalo 宿主桥接，并明确不替代真机验收。

## 9. 风险、未决问题与外部依赖

| 风险/依赖                      | 处理                                                                                                       |
| ------------------------------ | ---------------------------------------------------------------------------------------------------------- |
| 并发超卖或死锁                 | 数据库为事实源；稳定锁顺序、行锁、整批事务、幂等键和高并发集成测试                                         |
| 价格规则导致多折或负数         | 整数 VND、冻结叠加矩阵、每桶上限、折扣封顶、确定性选优和大量边界测试                                       |
| 购物车报价被误当最终金额       | 响应明确 `order_payable_vnd = null`/不可结算原因；M4 强制重算并存快照                                      |
| 搜索索引与发布内容漂移         | 同事务同步、可重建命令、校验任务和索引一致性测试；Redis 不作事实源                                         |
| `unaccent`/`pg_trgm` 权限      | PostgreSQL 17 将两者列为可信扩展，但目标数据库仍需安装文件和数据库 `CREATE` 权限；部署前显式探测并安全失败 |
| 搜索性能/中文分词有限          | P0 使用 FTS + trigram；记录查询计划与规模基线，达到阈值后另行评估 OpenSearch，不在 M3 提前引入             |
| 预留超时 worker 重复执行       | 数据库状态条件与动作键保证幂等；任务只传商城/预留 ID，失败可安全重试                                       |
| 真实销量、运费、券核销尚无事实 | 分别在 M4 订单/物流/下单事务接入；M3 不生成假数据或假成功                                                  |
| Zalo 真机与外部上线条件        | 继续作为上线前置条件；Web E2E 不能替代真实宿主、支付/物流或专业合规验收                                    |

预计可能新增 BullMQ 以运行库存预留过期扫描；在 M3.2 实施前必须核对固定版本、Node.js 24/Redis 8 兼容性、许可证和维护状态。若数据库轮询已能满足批准的可靠性与部署模型，则不为未来假设引入该依赖，选择结果记录在计划状态中。

PostgreSQL 设计依据：`unaccent` 用于移除变音符号，`pg_trgm` 提供相似度及索引支持；二者在 PostgreSQL 17 默认发行中属于可信扩展。行级锁在事务结束时释放，所有多行库存命令仍必须采用稳定顺序避免死锁。参考 PostgreSQL 17 官方文档：<https://www.postgresql.org/docs/17/unaccent.html>、<https://www.postgresql.org/docs/17/pgtrgm.html>、<https://www.postgresql.org/docs/17/explicit-locking.html>、<https://www.postgresql.org/docs/17/sql-createextension.html>。
