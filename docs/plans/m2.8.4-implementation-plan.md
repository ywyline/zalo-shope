# M2.8.4 可重复浏览器 E2E 与 M2 收口实施计划

> 状态：已完成（2026-07-20）
>
> 日期：2026-07-20
>
> 依赖：`REQUIREMENTS.md`、`AGENTS.md`、`docs/plans/m2.8-implementation-plan.md`、`docs/reports/m2.8.3-completion-report.md`

## 1. 目标与非目标

目标：

- 建立仓库内可重复执行的真实浏览器 E2E 项目，使用真实 API、PostgreSQL、Redis 与 MinIO，不以静态接口假数据冒充验收。
- 覆盖管理端密码 + TOTP 登录、双商城切换、目录隔离、三语文案、XLSX 模板下载与 dry-run 上传。
- 覆盖 Mini App 美妆/服装两套独立配置、越南语/中文/英文首页、商品详情、SKU 选择、错误恢复，以及常见 Android Chrome 与 iPhone Safari 视口。
- 将浏览器安装、E2E 执行、失败 trace/截图和报告归档纳入 GitHub Actions 独立门禁；本地提供明确命令和前置条件。
- 更新 M2.8/M2/P0 状态与完成报告，关闭“缺少可重复浏览器 E2E”保留项。

非目标：

- 不进入库存、购物车、促销、价格计算、订单、支付或物流等 M3+ 领域。
- 不把浏览器预览当作 Zalo 宿主真机、生产对象存储/CDN 或越南专业合规复核证据。
- 不持久化测试管理员密码、TOTP secret、访问令牌或浏览器会话；测试凭据仅在单次进程内随机生成并在 teardown 删除。
- 不建立脆弱的整页像素基线；本阶段使用可访问名称、业务哨兵、响应结果、主题属性和横向溢出断言。

## 2. 模块与文件

- `playwright.config.ts`：冻结浏览器项目、四个本地 web server、超时、重试、trace、截图和报告策略。
- `tests/e2e/`：全局数据准备/清理、管理端 E2E 和 Mini App E2E。
- `tests/fixtures/m26-browser-fixture.ts`：将既有双商城真实目录夹具改为可复用且仍兼容 CLI 的 setup/teardown，不复制第二套商品夹具。
- `package.json`、`pnpm-lock.yaml`、`.gitignore`、`README.md`：新增固定版本依赖、执行命令、浏览器产物忽略和本地说明。
- `.github/workflows/quality.yml`：安装 Chromium/WebKit、执行 E2E 并归档失败证据。
- 阶段计划与报告：同步 M2.8.4 完成状态、验证证据和仍受外部条件阻塞的项目。

依赖选择：固定 `@playwright/test@1.61.1`。该版本为 Apache-2.0、要求 Node.js 18+，兼容项目 Node.js 24；只安装本阶段使用的 Chromium 与 WebKit 浏览器。安装后必须更新锁文件并重新运行生产依赖审计。

## 3. 数据、接口与隔离

- 不新增数据库表、迁移、生产环境变量或 API；E2E 在现有 local/test schema 上运行部署迁移和幂等基础种子。
- 双商城目录夹具使用固定测试业务 ID、三语内容和 MinIO 测试媒体；每次 setup 先清除同一夹具，再创建，teardown 再清除，确保可重复运行。
- 管理员使用冻结的测试专用 ID 以便中断后定向清理，邮箱、密码和 TOTP secret 每次随机生成；只授予两个测试商城现有 `store-admin` 角色。teardown 删除会话、角色绑定、审计和管理员。
- 管理端断言先在美妆商城观察美妆哨兵并确认服装哨兵不可见，再切换服装商城执行反向断言；Mini App 由两个独立 Vite 进程分别注入 `VITE_STORE_CODE`。
- XLSX 浏览器用例只执行 dry-run，验证上传链路和零业务写入；正式导入数据库语义继续由 M2.8.3 API 集成测试覆盖。

## 4. 执行与 CI 契约

- 本地前置：`corepack pnpm infra:up`，再运行 `corepack pnpm test:e2e`，结束后运行 `corepack pnpm infra:down`。
- `test:e2e` 先构建共享包、部署已有迁移、执行幂等种子，再由 Playwright 启动 API、管理端和两个商城 Mini App 预览进程。
- 浏览器项目：管理端桌面 Chromium；Mini App Android Chromium；Mini App iPhone WebKit。测试串行使用同一受控夹具，避免并发修改共享商城数据。
- 本地和 CI 都拒绝复用已占用端口，防止连接到配置不明的旧服务；web server 退出由 Playwright 管理。
- CI 在基础设施集成测试后安装 Chromium/WebKit，运行 E2E；无论成功或失败都归档 HTML 报告，失败保留 trace 与截图。

## 5. 风险、兼容与回滚

| 风险                                                | 处理                                                                                  |
| --------------------------------------------------- | ------------------------------------------------------------------------------------- |
| 测试凭据或会话残留                                  | 运行时随机生成、只通过进程环境传给测试 worker，返回 teardown 闭包清理；日志不打印凭据 |
| 双商城夹具串店或清理过宽                            | 只操作冻结测试 ID；API/UI 双向哨兵断言；不删除非夹具业务编码                          |
| 浏览器测试偶发等待失败                              | 使用 Playwright web-first 断言、角色/标签定位和响应等待，不使用固定长睡眠             |
| 本地端口复用导致假通过                              | `strictPort` + `reuseExistingServer: false`，四个 server 使用固定独立端口             |
| CI 浏览器体积与时长                                 | 只安装 Chromium/WebKit；限制 worker、重试和 trace 策略，报告设置保留期                |
| 一次性 MinIO 初始化容器使 `compose --wait` 返回非零 | 基础服务健康等待与一次性 bucket 初始化拆分，保持重复执行幂等                          |

应用回滚只需移除 Playwright 配置、E2E 文件、脚本、CI 步骤和开发依赖；没有数据库迁移或生产数据需要回滚。测试运行中断时可再次运行 setup 清理固定目录夹具和测试专用管理员 ID，不影响正式业务数据。

## 6. 测试与验收

- 管理端：真实密码/TOTP 登录；双商城正反哨兵；越南语/中文/英文关键文案；模板下载；XLSX dry-run 结果；1440×900 无横向溢出。
- Mini App：两个商城独立名称、行业主题和商品；三语首页；商品详情与 SKU；API 首次失败后的可恢复提示；Android 与 iPhone 项目均无横向溢出。
- 自动化门禁：`corepack pnpm test:e2e`、`corepack pnpm verify`、`corepack pnpm test:integration`、`corepack pnpm audit --prod --audit-level high`、`docker compose config --quiet`、`git diff --check` 与敏感信息扫描。
- 最终检查测试管理员、浏览器进程、监听端口、Compose 容器和 Playwright 临时产物均按约定处理。

M2.8.4 完成仅关闭 M2 可重复浏览器 E2E 保留项；Zalo 真机、生产对象存储/CDN 与越南法律/税务/行业专业复核仍为上线前置条件，且不自动授权进入 M3。
