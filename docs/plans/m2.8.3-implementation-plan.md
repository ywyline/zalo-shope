# M2.8.3 受限 XLSX 商品导入与导出实施计划

> 状态：已完成；证据见 `docs/reports/m2.8.3-completion-report.md`
>
> 日期：2026-07-19
>
> 依赖：`REQUIREMENTS.md`、`AGENTS.md`、`docs/plans/m2.8-implementation-plan.md`、`docs/reports/m2.8.2-completion-report.md`

## 1. 目标与非目标

目标：

- 在现有 M2.7 严格 CSV 解析和逐商品导入服务上增加 `.xlsx` 模板下载与导入；两种格式共享同一组冻结字段、行级校验、dry-run、部分成功报告和只创建草稿规则。
- 增加当前授权商城的商品 XLSX 导出，以一行一个 SKU 的稳定只读结构输出商品、三语内容、类目、售价和规格选项；不暴露内部 ID、商城 ID、成本价、合规证件或对象存储信息。
- 在目录工作台商品区增加模板下载、单文件选择、dry-run/正式导入、行级结果和导出入口，并提供越南语、中文、英文状态文案。
- 对压缩包、工作表、公式、单元格类型、行列数和导出公式注入建立明确边界；商城隔离与 RBAC 仍由可信请求上下文、复合商城条件和数据库 RLS 共同保护。

非目标：

- 不实现库存导入、库存导出、批量库存调整或任何 M3 库存状态机。
- 不导入商品状态、媒体、合规资料、发布结果或商城标识；导入仍只创建 `DRAFT` 商品，已存在业务编码仍按现有规则报告失败。
- 不支持旧式 `.xls`、含宏 `.xlsm`、加密工作簿、多工作表导入、公式、图片、图表、外部链接或任意表头映射。
- 不新增异步导出任务或生产文件存储；超过同步导出上限时明确拒绝并提示缩小商城数据后重试，后续如有大规模导出需求另行规划。
- 不在本阶段建立可重复浏览器 E2E 脚本；M2.8.4 负责多商城、三语和多视口自动化验收。

## 2. 涉及模块与文件

- `apps/api/src/catalog-admin/product-import.ts`：抽取 CSV/XLSX 共用的严格行解析入口。
- `apps/api/src/catalog-admin/product-xlsx.ts`：XLSX ZIP 预检、模板生成、导入解析、导出编码和公式注入防护。
- `apps/api/src/catalog-admin/product-admin.controller.ts`、`product-admin.service.ts`：新增模板、导入和导出路由并复用既有授权、商城隔离、审计和导入事务。
- `apps/admin-web/src/main.tsx`、`catalog-workbench.tsx`、`styles.css`：支持二进制下载、multipart 上传和三语交互。
- `apps/api/src/catalog-admin/*.spec.ts`、`tests/integration/`：覆盖文件边界、公式、RBAC、dry-run、跨商城隔离与敏感字段不泄漏。
- `docs/api/openapi.m2.yaml`、阶段总计划、数据字典/权限说明和完成报告：同步契约、限制与验证证据。

依赖选择：API 运行时增加固定版本 `read-excel-file@9.3.2` 与 `write-excel-file@4.1.1`，仓库根测试工具以相同固定版本作为开发依赖读取/构造集成测试工作簿。两者为 MIT、支持 Node.js 18+，与项目 Node.js 24 运行基线兼容；安装后必须检查锁文件并运行生产依赖审计。若实际 API 无法满足公式或资源边界，停止使用该依赖而不是绕过安全限制。

## 3. API 与文件契约

新增：

- `GET /v1/admin/catalog/products/imports/template.xlsx`
- `POST /v1/admin/catalog/products/imports/xlsx?dry_run=true|false`
- `GET /v1/admin/catalog/products/exports/products.xlsx`

模板与导入工作簿只允许一个可见工作表，名称固定为 `products`。第一行必须与现有 CSV 的 20 个 `PRODUCT_IMPORT_COLUMNS` 完全一致；最多 2,000 个数据行、500 个 SKU/商品，压缩文件最多 2 MiB。单元格只接受空值、字符串、有限长度安全整数或布尔值；数值转换为十进制文本后继续走现有 Zod/业务校验。任何公式、隐藏/额外工作表、错误值、日期/对象单元格、非法 ZIP 路径、加密条目或超出解压边界的工作簿均整文件拒绝。

商品导出工作表名称固定为 `products`，按商品编码、SKU 编码稳定排序，一行一个 SKU，字段为：

- `product_code`、`product_status`、`product_enabled`、`brand_code`、`main_category_code`、`secondary_category_codes`
- `name_vi`、`selling_points_vi`、`description_vi`、对应 `zh` 和 `en` 三语字段
- `sku_code`、`sku_status`、`barcode`、`sale_price_vnd`、`market_price_vnd`、`weight_grams`、`sku_options`、`updated_at`

导出不包含 `cost_price_vnd`，因为当前目录权限没有独立财务读取能力。同步导出最多 10,000 个 SKU 行；超出时返回业务冲突而不是截断。文本单元格强制按文本写入；以 `= + - @` 开头的值增加可逆的前导单引号，避免在表格软件中被解释成公式。

## 4. 安全与资源边界

- 上传先检查文件名 `.xlsx`、MIME 和 ZIP 签名，再读取 ZIP 中央目录；最多 128 个条目、单条目最多 8 MiB、总声明解压大小最多 16 MiB、压缩比最多 100:1。
- ZIP 条目必须为相对规范路径，拒绝 `..`、绝对路径、反斜线、NUL、加密、分卷和重复规范路径；再检查工作簿 XML 中的工作表可见性、外部链接和任意 `<f>` 公式节点。
- 解析器只消费第一且唯一的 `products` 工作表，并严格限制 20 列；尾部空行不计入数据行，数据区中间空行保留行号并作为行级非法数据报告。
- 上传内容、单元格值和导出数据不写日志；错误响应只返回稳定错误码、行号和字段，不回显整行敏感内容。
- 所有接口继续要求现有 `store.catalog.read` 或 `store.catalog.manage` 权限；服务端不接受文件中的 `store_id`，所有查询和创建都从授权商城上下文派生。

## 5. 迁移、兼容与回滚

- 不新增数据库迁移，不改变商品、SKU、审计或权限状态机。
- CSV 路由与文件格式保持向后兼容；XLSX 解析完成后转换为同一个 `ParsedProductImportRow[]`，不复制第二套数据库写入逻辑。
- 应用回滚只需移除新增路由、UI 和两个依赖；已成功导入的草稿属于正常业务数据，不自动删除，可由有权限运营停用。
- 导出缓冲区只存在于请求生命周期，不上传对象存储、不落盘、不进入版本库。

## 6. 风险与处理

| 风险                          | 处理                                                                                 |
| ----------------------------- | ------------------------------------------------------------------------------------ |
| ZIP 炸弹或畸形 OOXML 消耗资源 | 在第三方解析器前检查中央目录、条目数、声明大小、压缩比和路径，再限制文件/行/列       |
| 公式或外部链接执行            | 导入 XML 级拒绝公式和外部链接；导出对危险文本加前导单引号并强制文本类型              |
| CSV 与 XLSX 校验漂移          | 两种格式共用冻结列和行到领域输入的唯一转换函数，单元测试做等价断言                   |
| 跨商城或敏感数据导出          | 授权商城条件 + RLS；集成测试放置第二商城哨兵并断言不出现；成本价和内部标识不进入投影 |
| 大商城同步导出占用内存        | 先计数并限制 10,000 SKU 行，生成后检查响应大小；超限显式失败，不静默截断             |
| 重复点击造成重复导入          | UI 在请求期间禁用提交；服务端继续由商品/SKU 唯一约束和逐商品事务防止重复业务编码     |

## 7. 测试与验收

- 单元：模板结构/越南语、CSV/XLSX 等价解析、非法表头、空文件、超大文件、超行列、公式、隐藏/额外工作表、外部链接、ZIP 路径/解压边界、非法单元格类型、导出顺序和公式注入。
- API 集成：未认证、读/写权限分离、商城上下文不匹配、dry-run 零写入、正式导入仅产生草稿、逐商品结果、导出第二商城哨兵不可见、成本与内部 ID 不出现、MIME/文件名正确。
- 管理端：模板/导出下载、文件类型限制、dry-run/正式导入确认、加载/成功/失败/部分成功和行级结果；三语文案与键盘标签完整。
- 静态门禁：`corepack pnpm verify`、基础设施启动后的 `corepack pnpm test:integration`、`corepack pnpm audit --prod --audit-level high`、格式、敏感信息和最终差异检查。

M2.8.3 完成不表示库存导入、M2.8.4 自动化 E2E、Zalo 真机、生产对象存储或越南专业合规复核完成，也不授权自动进入 M3。
