# M2.7 商品导入、版本与批量操作实施计划

> 状态：已实施，M2 总体验收有保留（已包含在用户批准的 M2.1-M2.7 范围内）
>
> 日期：2026-07-19
>
> 依赖：`REQUIREMENTS.md`、`AGENTS.md`、`docs/plans/m2-implementation-plan.md`、`docs/database/m2-data-dictionary.md`

## 1. 目标与非目标

目标：

- 提供商城范围、权限受控的 UTF-8 CSV 商品与 SKU 草稿导入模板、校验和逐行结果报告。
- 提供不可变商品发布版本的管理端列表与详情读取能力。
- 提供商品批量停用与主类目移动命令，逐项事务、乐观锁、审计和部分失败可解释。
- 同步 OpenAPI、数据字典、阶段计划与 M2 完成报告，并完成 M2 自动化门禁。

非目标：

- 不导入或修改库存，不自动提交审核、上架或批准合规资料，不实现 M3 搜索/计价能力。
- 不修改已发布版本快照；导入仅创建新商品草稿。
- 不接入生产对象存储、Zalo 真机或外部发布。
- 本阶段不引入 XLSX 解析依赖。`exceljs` 稳定版发布时间与当前维护节奏不满足本仓库新增依赖门禁；按 M2 已批准风险处理先交付 CSV，XLSX 保留为明确的后续兼容格式。

## 2. 模块与接口变化

- `packages/contracts`：新增导入查询、批量停用/移动命令及上限约束。
- `apps/api/src/catalog-admin/product-import.ts`：无依赖 CSV 解析、严格表头/UTF-8/大小/行数校验、字段转换和模板生成。
- `apps/api/src/catalog-admin/product-admin.controller.ts`：新增模板、CSV 导入、版本和批量命令路由；multipart 文件只进入内存且限制为 2 MiB、单文件。
- `apps/api/src/catalog-admin/product-admin.service.ts`：新增导入预检/执行、版本读取和逐项批量事务；复用可信商城上下文、RLS、RBAC 与审计。
- `tests`：新增解析器单元测试和 API 集成测试，覆盖越权、跨商城、乐观锁、部分失败、版本不可变及无库存字段。
- `docs/api/openapi.m2.yaml`、`docs/database/m2-data-dictionary.md`：同步契约和实现规则。

新增接口：

- `GET /v1/admin/catalog/products/imports/template.csv`
- `POST /v1/admin/catalog/products/imports/csv?dry_run=true|false`
- `GET /v1/admin/catalog/products/{productId}/versions`
- `GET /v1/admin/catalog/products/{productId}/versions/{version}`
- `POST /v1/admin/catalog/products/batch/disable`
- `POST /v1/admin/catalog/products/batch/move`

## 3. 导入格式与事务语义

- 模板固定列：商品编码、品牌编码、主/辅助类目编码、三语名称/卖点、SKU 编码、条码、整数 VND 价格、重量和规格选项。
- 同一商品的多个 SKU 使用多行表达；商品级字段必须在组内完全一致。规格使用 `attribute_code=option_code`，多个规格以 `|` 分隔。
- 文件必须是 UTF-8（允许 BOM）和严格 CSV；拒绝 NUL、非法引号、重复/未知/缺失表头、超限列、2 MiB 以上文件及 2,000 行以上数据。
- `dry_run=true` 只执行格式、字段、引用、商城归属、模板规格和唯一性检查，不写数据库。
- 实际导入按商品分组；每个商品及其全部 SKU 在一个事务中创建，组内任一行失败则该商品不产生部分数据。不同商品允许部分成功，响应逐行返回 `IMPORTED`、`VALIDATED` 或 `FAILED` 及稳定错误码。
- 导入只接受业务编码，不接受 `store_id`、状态、库存、媒体、合规审批或发布版本字段；商城归属仅来自认证后的服务端上下文。

## 4. 批量操作与版本规则

- 批量命令最多 100 个不重复商品，每项携带 `expected_version`，并要求固定二次确认码。
- 停用可作用于未停用商品，设置 `status=DISABLED`、`enabled=false` 并递增乐观锁；历史发布版本保留。
- 主类目移动只允许 `DRAFT`/`UNPUBLISHED`。目标必须是同商城启用末级类目且绑定启用主属性模板；若模板变化且商品已有 SKU/属性值则拒绝，避免静默破坏规格。
- 每个商品独立事务并写独立审计；不存在、跨商城、版本冲突或状态冲突只影响对应项。
- 版本列表只返回元数据，版本详情返回该商城内不可变快照的运营视图；均要求 `store.catalog.read`，响应递归移除成本价，完整快照仍只保留在数据库。

## 5. 数据、迁移、兼容与回滚

- 不新增数据库表、字段或迁移，复用 M2.2 的商品、SKU、版本、复合约束和强制 RLS。
- API 为 `/v1` 向后兼容新增，现有 M1/M2 调用方不受影响。
- 应用可回滚到 M2.6；已导入草稿和审计记录保留，可由受权运营停用。批量停用不提供未审计恢复；误操作按后续受审命令向前修复。
- CSV 导入不写外部对象存储，不产生需要清理的文件对象。

## 6. 风险与外部依赖

- CSV 可能被公式注入或编码欺骗：导入不执行公式，模板与错误报告不回显为可执行公式；严格 UTF-8、表头和控制字符校验。
- 大文件可能造成内存/数据库压力：控制器 2 MiB、2,000 行、每商品 500 SKU、每批量命令 100 项；不做无界并行。
- 并发更新可能覆盖运营修改：所有变更使用当前版本检查，数据库唯一约束作为最终保护。
- XLSX 暂不支持：不引入维护状态不足或发布链不明确的依赖；后续选择依赖时必须重新审查许可证、供应链、Node 24 与解压资源限制。
- Zalo 真机、生产对象存储/CDN 和自动化浏览器 E2E 仍是既有外部限制，本阶段不宣称补验。

## 7. 测试与验收

- 单元：BOM/越南语/引号、畸形 CSV、表头、整数 VND、行数/列数、商品分组和规格解析。
- API/集成：模板权限、dry-run 零写入、真实导入、逐行失败、重复编码、跨商城引用、RBAC、批量部分失败、乐观锁、模板不兼容移动、版本列表/详情与快照不可变。
- 安全：multipart 大小/类型、未知字段、库存字段、跨商城 UUID/编码、未授权读取和错误响应不泄漏内部信息。
- 门禁：`corepack pnpm test:unit`、`corepack pnpm test:integration`、`corepack pnpm verify`、`corepack pnpm audit --prod --audit-level high`、`git diff --check` 和敏感信息/最终差异审查。
- M2 收口报告必须逐项记录已运行证据、未运行项、已知限制、回滚方法，并明确不自动进入 M3。
