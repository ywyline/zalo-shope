## 1. 目标与非目标

目标：

- 在现有 PC 管理端新增商城范围的目录工作台，覆盖品牌、两级类目、属性模板、商品与合规五个区域。
- 品牌支持三语创建/编辑、推荐、排序与停用；类目支持三语创建/编辑、树形浏览、排序、停用和主属性模板绑定。
- 属性模板支持定义构建、有限选项、创建、版本查看和带二次确认的激活。
- 商品支持列表/筛选、三语草稿创建、SKU 摘要、非规格属性编辑、提交门禁结果和带二次确认的发布/停用。
- 合规区域展示当前商城启用要求和最近记录的脱敏读模型，并允许有权限且满足审核人分离规则的管理员批准或驳回。
- 所有界面提供越南语、中文和英文文案；商城切换立即清空旧商城状态，并提供加载、空、失败、无权、冲突、成功和重试反馈。
- 完成桌面、窄屏、键盘焦点和真实 API 浏览器验收，不用静态假数据冒充集成。

非目标：

- 不新增或修改库存、价格计算、购物车、订单、支付、物流或 M3 状态机。
- 不实现 SKU 成本价编辑。现有权限模型没有财务扩展权限，列表也有意不返回成本价；不能通过全量替换静默清空已有成本。
- 不实现媒体二进制上传、裁剪或生产对象存储管理；现有媒体 API 和生产 provider 前置条件保持不变。
- 不实现 XLSX、商品导出或库存导入，它们属于 M2.8.3。
- 不在本阶段建立可重复浏览器 E2E 项目脚本，它属于 M2.8.4；本阶段仍执行真实浏览器验收并如实记录。

- `apps/admin-web/src/catalog-workbench.tsx`：新增五区目录工作台、表单、属性编辑器、门禁与审核交互。

- `tests/integration/`、`packages/contracts/src/index.spec.ts`：覆盖查询边界、RBAC、商城隔离、脱敏和 UI 所依赖的响应形状。

## 3. API 与数据变化

新增：

- `GET /v1/admin/compliance/overview?store_id={storeId}&limit=50&status=PENDING_REVIEW`

查询最多返回 100 条最近合规记录，可按状态和商品 UUID 过滤；要求 `store.compliance.read`。响应包含：

- 启用合规要求的编码、文档类型、类目范围、阻断标志和版本。
- 合规记录的状态、提交/审核/到期时间、媒体数量、关联商品编码与越南语名称。
- 证件编号只返回掩码，不返回媒体对象键、下载地址、审核资料正文或其他商城标识。

类目列表向后兼容增加 `category_attribute_templates` 关系，只包含同商城绑定和模板版本/模板元数据。没有数据库迁移，不改变现有表、状态机或权限种子。

## 4. UI 工作流

- 工作台首次加载并行读取品牌、类目、模板、商品与合规概览；任一请求失败时不混用部分商城旧数据，显示可重试错误状态。
- 三语输入使用统一字段组；越南语明确必填，中文/英文可完整维护。展示语言缺失时只在读取视图回退越南语，不把回退值冒充已填写翻译。
- 编辑命令始终携带服务端返回的 `expected_version`；409 冲突提示用户重新加载，不在前端强行覆盖。
- 高风险激活、发布、停用和合规审核使用可访问确认对话框；确认值必须匹配对象业务编码或明确决策词。
- 商品属性编辑复用 M2.8.1 判别类型：文本按语言、有限选项按 code、整数/定点小数/布尔/日期按严格输入控件；保存时全量发送当前编辑器值。
- 商城切换以 `store.id` 重新挂载工作台并清空选中对象、表单、通知和缓存；所有请求头仍由内存访问令牌和可信商城上下文生成。

## 5. 迁移、兼容与回滚

- 不新增数据库迁移；API 均为 `/v1` 向后兼容新增或响应字段扩展。
- 管理端只调用现有受审命令；应用回滚不会改变已保存的品牌、类目、模板、商品、属性、合规记录或审计。
- 合规概览可通过回滚 API 路由关闭，不影响提交/审核状态机；审计和审核记录不删除。
- UI 失败不会保留跨商城数据或把未确认表单自动提交；用户刷新后从服务端重新读取当前版本。

## 6. 风险与处理

| 风险                     | 处理                                                                               |
| ------------------------ | ---------------------------------------------------------------------------------- |
| 前端状态在商城切换后泄漏 | `key={store.id}` 重新挂载，加载前清空数据，API 仍由 RLS 和复合商城条件保护         |
| 过期版本覆盖运营修改     | 每个更新携带 `expected_version`；409 显示冲突并要求重新读取                        |
| 合规编号或文件信息泄漏   | 服务端只生成掩码和媒体计数，不返回对象键、文件 URL 或原始供应商内容                |
| 审核人审核自己提交的资料 | UI 不作可信判断；服务端继续执行提交人/审核人分离和状态检查                         |
| 模板表单产生非法组合     | UI 按用途/类型约束控件，共享 Zod 契约和服务端再次校验                              |
| 全量 SKU 编辑丢失成本    | 本阶段只展示安全 SKU 摘要，不开放缺少财务权限设计的替换表单                        |
| 浏览器验证不可重复       | 记录真实交互证据和清理结果；M2.8.4 再固化自动化脚本，不把本轮手工证据冒充 E2E 项目 |

## 7. 测试与验收

- 契约单元：合规查询 limit、状态、商品 UUID、未知字段和字符串布尔/数字边界。
- API 集成：合规概览只读 RBAC、跨商城隔离、证件编号掩码、媒体/对象信息不泄漏、过滤和有界结果；类目模板绑定投影保持同商城。
- 静态门禁：`corepack pnpm verify`、`corepack pnpm test:integration`、生产依赖审计、Compose、格式、敏感信息与最终差异检查。
- 浏览器桌面：密码 + MFA、五个区域、三语切换、品牌/类目/模板/商品真实写入、属性保存、合规列表/审核状态、冲突/失败/空状态和商城切换。
- 浏览器响应式：至少 1440×900、900×900 和 390×844；检查无水平页面溢出、对话框可见、表单标签关联、Tab 焦点顺序和可见焦点。
- 浏览器验收使用一次性 local/test 管理员和数据；完成后清理管理员、会话、审计及本轮创建的业务数据，并确认无前缀残留。

M2.8.2 完成不表示 XLSX/导出、自动化 E2E、Zalo 真机、生产对象存储或越南专业合规复核完成，也不授权自动进入 M2.8.3 或 M3。

完成证据见 `docs/reports/m2.8.2-completion-report.md`。后续阶段必须继续单独执行计划、门禁和汇报。
