# M2.8 M2 验收缺口收口计划

> 状态：M2.8.1-M2.8.4 已完成
>
> 日期：2026-07-20
>
> 依赖：`REQUIREMENTS.md`、`AGENTS.md`、`docs/plans/m2-implementation-plan.md`、`docs/reports/m2-completion-report.md`

## 1. 背景与批准边界

M2.1-M2.7 已通过当前自动化门禁，但 M2 总结仍记录商品属性值编辑、目录管理端 UI、XLSX/导出和可重复浏览器 E2E 等验收保留。原 M2 计划明确禁止自动进入 M3；当前任务要求按严谨工作流继续下一阶段，但没有明确接受这些结转风险。因此本计划将下一阶段解释为 M2 内部收口，不进入库存、购物车、促销或价格计算等 M3 高风险领域。

本计划按独立子阶段推进，每个子阶段通过测试和文档门禁后才进入下一项：

1. M2.8.1：商品非规格属性读取、编辑和发布门禁闭环。
2. M2.8.2：品牌、类目、属性模板、商品和合规管理端工作台。
3. M2.8.3：受限 XLSX 导入与商品导出。
4. M2.8.4：可重复浏览器 E2E、双商城/三语/视口验收和 M2 总结更新。

Zalo 真机、生产对象存储/CDN 与越南行业法律复核依赖外部环境或专业人员，只能保留为上线前置条件，不能用本地假实现关闭。

## 2. M2.8.1 目标与非目标

目标：

- 提供商品属性编辑器读取接口，返回商品版本、可编辑状态、固定模板定义、启用选项和当前非规格属性值。
- 提供非规格属性值全量替换命令，覆盖文本、整数、定点小数、布尔、日期和选项类型。
- 强制商城隔离、`store.catalog.read/manage` 权限、草稿状态、乐观锁、模板归属、类型、选项状态和多值约束。
- 文本属性使用 `vi`/`zh`/`en` 明确标识语言；必填文本属性只有存在非空越南语值时才满足发布门禁。
- 变更递增商品版本并写入前后快照审计，响应不暴露其他商城或内部错误细节。

非目标：

- 不修改规格属性；`SPECIFICATION` 继续通过 SKU 选项接口维护。
- 不在本子阶段修改商品基础资料、类目、品牌、上下架状态或合规审批。
- 不新增管理端 UI、XLSX、导出、库存、价格、购物车、搜索或第三方集成。
- 不迁移历史发布快照；新属性只在后续发布时进入新的不可变快照。

## 3. 模块与接口变化

- `packages/contracts/src/catalog.ts`：新增严格的属性值判别联合、全量替换命令和重复值校验。
- `apps/api/src/catalog-admin/product-admin.controller.ts`：新增读取与替换路由。
- `apps/api/src/catalog-admin/product-admin.service.ts`：新增编辑器投影、校验、事务替换、版本递增、审计和发布门禁修正。
- `tests/integration/` 与 `packages/contracts/src/index.spec.ts`：新增契约、权限、跨商城、状态、并发和发布门禁回归测试。
- `docs/api/openapi.m2.yaml`、`docs/database/m2-data-dictionary.md`、`docs/security/m2-permission-matrix.md`：同步接口、数据语义和权限。

新增接口：

- `GET /v1/admin/catalog/products/{productId}/attributes`
- `PUT /v1/admin/catalog/products/{productId}/attributes`

替换命令使用判别联合：

- `TEXT`：`attribute_code`、`data_type`、`locale`、`value`。
- `INTEGER`：`attribute_code`、`data_type`、安全整数 `value`。
- `DECIMAL`：`attribute_code`、`data_type`、最多 16 位整数和 8 位小数的十进制定点字符串 `value`。
- `BOOLEAN`：`attribute_code`、`data_type`、布尔 `value`。
- `DATE`：`attribute_code`、`data_type`、真实日历日期 `YYYY-MM-DD` 字符串 `value`。
- `OPTION`：`attribute_code`、`data_type`、`option_code`。

所有对象严格拒绝未知字段。单值文本允许每种语言各一项；其他单值属性只能出现一次。多值属性允许多个不同值，但拒绝完全重复项。请求最多 500 个值。

## 4. 数据、事务与兼容

- 复用 `product_attribute_values` 的类型列、复合外键、单值检查、强制 RLS 和商品版本；本子阶段不新增迁移。
- 读取只使用认证后的商城上下文和复合商城条件，不接受请求体中的商城归属。
- 替换事务先读取商品及固定模板，再验证全部输入；任一值非法则整批不写入。
- 删除并重建该商品的非规格属性值后，以 `store_id + product_id + expected_version` 条件更新商品并递增版本；条件未命中视为并发冲突，事务回滚。
- 规格属性值不在替换范围内；即使数据库中存在异常的规格级商品属性行也不由本命令静默删除。
- API 是 `/v1` 向后兼容新增。应用回滚后数据仍符合现有 schema；误修改通过下一次受审替换向前修复，不删除审计或已发布版本。

## 5. 校验与发布规则

- 属性必须属于商品创建时固定的模板版本，且用途不能是 `SPECIFICATION`。
- 请求 `data_type` 必须与定义一致；选项必须属于该定义并处于 `ACTIVE`。
- 文本先去除首尾空白且不得为空；整数必须为 JavaScript 安全整数；小数禁止指数、浮点数和超出 `DECIMAL(24,8)` 的值；日期必须能按 UTC 往返为同一天。
- 定义的 `multiple=false` 时执行单值规则；`multiple=true` 时允许不同值重复语言，但仍禁止完全重复值。
- 必填非文本属性存在任一合法值即可满足发布门禁；必填文本属性必须存在非空 `vi` 值。必填规格属性仍由至少一个启用 SKU 的选项选择满足。
- 当前模板 `validation_rules` 尚未冻结正式结构，M2.8.1 不对任意 JSON 静默解释；格式和业务类型边界由上述固定契约保证。后续如引入最小值、正则等规则，必须先版本化规则 schema 并补迁移/兼容计划。

## 6. 风险、回滚与外部依赖

| 风险                                | 处理                                                         |
| ----------------------------------- | ------------------------------------------------------------ |
| 删除重建造成并发覆盖                | 同事务乐观锁条件更新，失败时整体回滚；审计保存前后快照       |
| 用错误模板或跨商城 ID 写值          | API 只接受业务 code；定义、选项和商品均在可信商城事务中解析  |
| 小数经二进制浮点失真                | API 只接受定点十进制字符串，直接写入 Prisma Decimal 兼容字段 |
| 文本多语义混淆                      | 文本强制 locale；越南语必填单独进入发布门禁                  |
| 任意 validation JSON 被误认为已执行 | 明确不解释未版本化规则，文档记录后续冻结要求                 |

不涉及数据库迁移，应用代码可回滚到 M2.7。已经保存的属性值继续由既有表和 RLS 保护；不得回退或删除审计/发布版本。

## 7. 测试与验收

- 契约单元：所有数据类型、未知字段、小数/日期边界、重复单值和完全重复多值。
- API/集成：读取投影、合法全量替换、版本递增、前后审计、只读拒绝、跨商城 404、规格属性拒绝、错误类型/停用选项拒绝、非草稿拒绝和版本冲突回滚。
- 发布回归：必填文本只有中文/英文时仍返回 `ATTRIBUTE_REQUIRED`；补齐越南语值后该问题消失。
- 数据库：确认所有写入带 `store_id`，现有 one-value check、复合外键和 RLS 继续生效。
- 门禁：`corepack pnpm test:unit`、`corepack pnpm test:integration`、`corepack pnpm verify`、`corepack pnpm audit --prod --audit-level high`、`git diff --check` 和敏感信息/最终差异审查。

M2.8.1 完成只表示属性 API 缺口关闭，不代表 M2 总体验收无保留，也不授权自动进入 M2.8.2 或 M3。
