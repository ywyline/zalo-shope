# M2.2 数据库迁移与隔离完成报告

日期：2026-07-17

结论：M2.2 已完成。M2 商品内容关系模型、扩展迁移、商城隔离、关键数据库不变量和 local/test 基础种子均已落地；下一阶段按批准计划应为 M2.3 品牌、类目与属性模板后台 API（原报告误写为媒体基础设施，现已纠正）。

## 已交付

- Prisma schema 新增品牌、类目、属性模板及版本、SPU、SKU、媒体、合规、商品版本和页面装修关系模型。
- `20260717141931_m2_catalog_content` 扩展迁移新增 29 张商城表及所需枚举、索引、检查约束、复合外键、RLS、触发器和 runtime role 授权，不修改或删除 M1 字段。
- 每张 M2 商城表均启用并强制 RLS，runtime 缺少 transaction-local `app.store_id` 时读取为空、写入失败；`store_id` 更新由触发器拒绝。
- 跨商城品牌、类目、模板、商品、SKU、媒体、合规和页面关系由包含 `store_id` 的复合外键保护。
- 已激活属性模板及其定义/选项、已审核合规记录、已发布/撤回商品版本和页面版本及其模块内容不可原地修改。
- SKU 规格值必须来自 `SPECIFICATION` 属性，数据库按稳定属性编码排序重算组合 key 和 SHA-256；主类目不能重复绑定为辅助类目。
- local/test 种子幂等创建两商城可识别的根/子类目、基础属性模板/规格选项、七项 M2 商城权限，并只更新可识别的 `store-admin` 系统角色；不创建商品或合规批准假数据。
- 提供仅限无真实 M2 数据的 local/test `down.sql`；生产及产生媒体、审核或版本数据后的环境只允许向前修复。

## 验证证据

- 测试先行：新增 M2 集成测试最初因 `brands` 等关系不存在而失败，迁移完成后通过。
- M2 专项数据库测试：7/7 通过，覆盖无上下文默认拒绝、跨商城同码隔离、种子/权限、29 表逐表强制 RLS 与归属不可变、SKU 组合重算、跨店复合外键、类目结构及最终记录不可变。
- 全量集成回归：5 个测试文件、25/25 通过，包含既有 M1 身份、RBAC、审计和 API 回归。
- 隔离数据库 `zalo_shop_m2_final_audit`：空库三阶段迁移通过；第二次 `migrate deploy` 无待执行迁移；种子连续执行两次成功；数据库与 Prisma schema 无差异。
- 同一隔离库人工执行 M2 `down.sql` 后，M2 `brands` 已移除而 M1 `stores` 保留；验证后已删除隔离库。
- 本地测试库在执行强制 reset 前已核对为项目 Compose 的 `localhost:5432/zalo_shop`，只包含固定测试种子和随机测试记录；重建后重新种子。

## 风险与限制

- Prisma 6 对 RLS 和检查约束不能完整建模；这些约束保留在经审查的 SQL 迁移中，并由集成测试逐项保护。后续迁移必须执行 schema drift 检查。
- `down.sql` 不会删除共享的 `pgcrypto` 扩展，避免影响同数据库集群的其他功能；这不影响应用回滚。
- M2.2 不包含媒体对象上传、商品 CRUD API、发布编排或 UI；这些分别属于 M2.3 至 M2.6。
- Zalo Mini App 真机验收继续作为已接受的结转风险，不因数据库阶段完成而关闭。

## 下一阶段门槛

M2.3 必须先覆盖品牌、类目与属性模板后台 API 的请求体商城归属拒绝、RBAC、跨商城父级/模板、两级类目、乐观版本、审计和激活版本不可变，再实现对应服务。媒体对象键隔离、MIME/大小/checksum、SVG 清洗和失败状态属于 M2.4。
