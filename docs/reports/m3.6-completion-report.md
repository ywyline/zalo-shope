# M3.6 会员购物车、可售投影与移动端体验完成报告

> 状态：已完成
>
> 日期：2026-07-22
>
> 范围：`docs/plans/m3-implementation-plan.md` 的 M3.6；不包含 M3.7 全阶段并发/安全收口，也不包含 M4 地址、结算、订单、COD、支付或物流。

## 1. 交付结果

- 新增商城/会员隔离的 `GET /v1/cart`、按 SKU 设置最终数量的幂等 `PUT`、带行乐观锁的 `PATCH` 与 `DELETE`；支持数量、选中、删除和同商品 SKU 替换。所有端点接受 `locale`，默认越南语并按越南语回退商品、规格与媒体文案。
- 购物车每次读取和修改都重新加载商品发布状态、SKU 状态、默认履约仓可售量与当前促销事实，并复用可信价格服务返回选中且无阻断行的临时报价。
- 商品下架、SKU 停用、零库存与库存不足为阻断问题；价格与促销变化为非阻断提示。加入购物车不创建库存预留，不核销优惠券，不生成订单或最终应付金额。
- 公开目录、商品详情与搜索结果增加默认履约仓实时可售量；当前生效的同商城 ITEM/ORDER 活动按品牌、类目、商品、SKU 或 STORE 精确目标返回本地化摘要，`on_promotion` 使用真实投影筛选。
- Mini App 商品详情接入真实购物车，提供数量、加入、购物车角标与底部导航；购物车页支持选择、数量、失效规格替换、删除、变化提示和服务端金额摘要。语言切换会重新加载同一购物车的本地化展示。
- 未登录时进入 Zalo 身份流程；Web 预览和授权失败保留可恢复入口。访问令牌仍只保存在 React 内存中，不写 LocalStorage/SessionStorage。
- 客户端拒绝在购物车初次加载完成前用陈旧数量执行“加入”，以购物车聚合版本阻止乱序响应回退；401、网络失败、删除后刷新失败和 409 冲突均保留重新登录或重试入口。

## 2. 安全与一致性

- 购物车 API 只接受商城绑定会员令牌与 `X-Store-Code`；所有查询/写入在强制 RLS 的商城事务中执行，请求体不接受 `store_id`、单价、折扣或应付金额。
- 首次活动车创建与写操作锁定会员/购物车事实；唯一冲突和 PostgreSQL 事务写冲突（P2002/P2034）采用有界整事务重试。相同 PUT 不累加数量，也不重复递增行或聚合版本。
- PATCH/DELETE 使用 `expected_version`；跨会员或跨商城行 ID 按当前主体资源不存在处理。SKU 替换仅允许当前行同一商品的同商城活动 SKU，目标行已占用时返回稳定冲突。
- 购物车报价运行在 `REPEATABLE READ` 事务中，价格/促销指纹仅用于变化提示；M4 必须重新计价、校验优惠券、锁库存并保存订单快照。
- 促销指纹排除只命中其他 SKU 的 `TARGET_MISMATCH`，避免无关活动造成虚假“促销变化”，同时保留当前行适用/未满足规则的变化检测。
- 活动摘要只展示当前商城 active root 的 published active version 与有效时间窗，不跨店、不递归扩张类目，也不包含 COUPON/SHIPPING 规则；摘要不是价格承诺。
- HTTP 日志脱敏同步覆盖 pino-http 重命名后的 `request`/`response` 字段，Authorization、Cookie、Zalo/refresh token 与响应 Cookie 均有真实 HTTP 哨兵回归。

## 3. 验证结果

- `corepack pnpm verify`：格式、lint、四应用类型检查、21 个单元测试文件共 128 项、生产构建与 Prisma schema 校验全部通过；其中新增购物车促销指纹和 HTTP 凭据脱敏回归。
- `corepack pnpm test:integration`：19 个文件、92 项通过；M3.6 购物车 3 项覆盖 PUT 幂等、会员/商城隔离、三语展示、乐观锁、库存/下架/价格变化与错误商城 Header，目录/搜索投影 2 项覆盖实时库存、活动/暂停对照、`on_promotion` 正反筛选、跨商城和三语摘要。数据库集成文件顺序执行，文件内部的库存/幂等并发断言保持并发，避免共享测试库夹具互相污染。
- `E2E_API_PORT=3100 corepack pnpm test:e2e`：11/11 通过，包括管理端 Chromium 3 项、Pixel 7 Chromium 4 项、iPhone 13 WebKit 4 项；覆盖双商城三语目录/搜索、实时库存、促销工作台以及购物车 Web 预览可恢复登录状态。开发中修正过一次 Playwright 非精确标题定位；最终全量重跑通过，之前观察到的单次 Vite `ECONNRESET` 未复现。
- `corepack pnpm audit --prod --audit-level high` 无已知漏洞；`docker compose config --quiet`、OpenAPI M2/M3 YAML 解析与本地 `$ref` 完整性检查、`git diff --check` 均通过。

## 4. 数据与兼容性

- 本阶段复用 M3.2 已迁移的 `carts`、`cart_items`、复合外键、唯一索引、检查约束与强制 RLS，不新增数据库迁移。
- 目录/详情新增的 `available`、`available_quantity`、`promotion_summary`，以及购物车 `locale` 查询和展示字段均为向后兼容扩展；未传语言仍为 `vi`。展示媒体签名 URL 不参与购物车事实或价格计算。
- 购物车一旦产生真实数据，不执行 M3 foundation `down.sql`；应用回滚时保留 schema 与购物车事实，只允许向前修复。

## 5. 限制与下一阶段

- Web E2E 不能替代 Zalo 宿主真机；真实 Zalo 凭据、生产对象存储/CDN、支付/物流沙箱和越南专业合规复核仍未验证。
- M3.6 浏览器用例验证 Web 预览下的可恢复登录状态；Zalo 宿主中的真实身份交换及认证后完整加入/数量/选择/SKU 替换/删除/失效/重算矩阵按批准计划在 M3.7 收口，不把 Web 模拟声明为真机验收。
- M3 购物车不提供运费、偏远附加费、优惠券核销、库存预留或最终订单金额；这些必须在 M4 的同一可信下单事务中完成。
- 美妆与服装生产构建仍必须分别显式注入正确的 `VITE_STORE_CODE` 和对应 Zalo 配置；本地默认 `beauty-local` 只用于兼容当前开发构建，不能作为服装生产部署配置。
- 下一阶段按批准计划进入 M3.7：完整并发、安全、两个商城三语 Chromium/WebKit 购物车流程、迁移演练与 M3 阶段收口。M3.7 完成仍不自动进入 M4。
