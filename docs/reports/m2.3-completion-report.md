# M2.3 品牌、类目与属性模板后台 API 完成报告

日期：2026-07-17

结论：M2.3 已完成。品牌、两级类目和属性模板不可变版本的商城范围后台 API 已接入 M1 身份/RBAC/审计链路；M2.4 商品、SKU、媒体与合规后台 API 尚未开始。

## 已交付

- 新增 `CatalogAdminController` 与 `CatalogAdminService`，所有命令从认证主体、`X-Store-Code` 和 `store_id` 查询参数交叉确认商城，不接受请求体 `store_id`。
- 品牌支持列表、创建、三语资料更新、推荐/排序、乐观版本和停用；不暴露物理删除。
- 类目支持含停用项的两级树、根/子类目创建、排序、移动、停用、乐观版本及已激活模板版本绑定/解绑；拒绝自身循环、三级结构、跨商城父级及跨商城模板。
- 属性模板支持列表、创建首个草稿、创建下一草稿版本、替换草稿定义/选项和激活；激活需要 `store.catalog.publish`，已激活版本及其定义/选项继续由数据库触发器保护。
- 创建与变更分别写入 `catalog.brand.*`、`catalog.category.*` 和 `catalog.attribute_template.*` 审计事件，保留前后版本并经日志脱敏。
- DTO 与 OpenAPI 同步覆盖品牌、类目、属性定义/选项、模板草稿、乐观版本和激活命令。

## 数据库向前修复

M2.3 回归测试发现 M2.2 通用终态触发器直接比较多个 PostgreSQL 枚举，模板从 `DRAFT` 激活时会错误解析其他状态枚举。新增 `20260717151500_m23_finalized_status_text` 向前迁移，按表分支并转为文本比较；不改变表结构或业务数据。该安全修复不提供恢复缺陷实现的反向迁移，生产环境只允许向前应用。

## 测试覆盖

- 契约单元测试：后台请求体拒绝 `store_id`，属性规格必须使用选项类型，越南语名称和业务编码按冻结规则校验。
- M2.3 API 集成测试：6 个场景覆盖品牌 CRUD/审计、只读 RBAC、请求体商城篡改、两级类目/循环/跨店父级、乐观冲突、模板激活不可变和跨店模板拒绝。
- 测试清理由数据库 owner 仅对本测试记录在单事务中临时关闭触发器；runtime 角色无此权限，业务 API 始终受不可变触发器约束。
- local/test 数据库从空库重建并依次应用 M1、M2.2 和 M2.3 四个迁移，再重新种子成功。

## 验证证据

- `corepack pnpm verify`：通过；格式、lint、类型检查、47/47 单元测试、全部应用生产构建和 Prisma schema 校验均成功。
- `corepack pnpm test:integration`：6 个测试文件、31/31 场景通过，包括既有 M1、M2.2 数据库安全及新增 M2.3 API 回归。
- 重复 `migrate deploy`：无待执行迁移；迁移目录与 Prisma schema 漂移检查结果为 `No difference detected.`。
- 最终 local/test 数据库再次从空库应用四个迁移并种子；核对仅有 2 个固定商城、2 个固定模板和 0 个品牌，无本轮随机 API 测试记录残留。

## 兼容与回滚

- 新增 API 路径和 DTO，不破坏 M1 接口；没有新增运行时环境变量或外部依赖。
- 品牌和类目不提供 `DELETE`，历史引用继续由复合外键及停用策略保护。
- API 代码可回退；数据库触发器修复应保留。尚未产生 M2 数据的 local/test 环境可按完整 M2 回滚流程处理，真实数据环境仅允许向前修复。

## 已知限制与下一阶段

- 本阶段不含商品、SKU、媒体上传、合规、管理端 UI 或买家目录 UI；这些分别属于 M2.4-M2.6。
- 属性模板目前要求整版替换草稿定义，适合版本化编辑；批量逐项结果与导入属于后续阶段。
- Zalo Mini App 真机验收继续作为用户已接受的结转风险，不因后台 API 完成而关闭。
- 下一阶段为 M2.4：先冻结并测试商品/SKU/媒体/合规命令的事务、商城归属、金额、媒体安全、权限分离和上架门禁，再实施 API。
