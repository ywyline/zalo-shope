## 3. 关键冻结决策

1. 现有量、预留量是库存事实，可售量只派生；无余额不自动补“无限库存”。
2. P0 买家只读取每商城一个启用默认履约仓；schema 保留多仓，不实施 P1 分配/调拨。
3. 计价槽为 `ITEM -> ORDER -> COUPON`，`SHIPPING` 只判断资格；跨槽叠加要求双方声明，同槽只选最大 VND 优惠。
4. 价格请求只含 SKU code、数量和可选券码；`order_payable_vnd` 在 M3 必须为 `null`，M4 重新加载地址/运费/事实并计价。
5. 搜索使用 PostgreSQL FTS + `unaccent` + `pg_trgm` 的受控投影；Redis 不作为搜索事实源。
6. 购物车保存加入价仅用于变化提示，读取/修改均重新加载发布、SKU、库存和促销事实；购物车不预留库存。
7. 管理端请求体不接受 `store_id`；商城由受信 Header、令牌授权和强制 RLS 三层共同约束。
8. M3.2 迁移只能按冻结字典向后兼容新增；字段、状态或接口发生实质变化时必须先更新契约并记录原因。
