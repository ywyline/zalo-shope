# M3.3 仓库、库存、预留原语与管理端工作台完成报告

>
> 日期：2026-07-20

## 1. 交付结果

- 新增商城隔离的仓库列表、创建、三语编辑、启停和默认履约仓切换；使用乐观锁、数据库唯一约束和前后快照审计，默认仓替换同时审计被降级仓库。
- 新增库存余额/流水分页、SKU/仓库/有货筛选、受审差量调整和明确二次确认。余额不提供直接编辑接口；调整在同一事务写动作、余额、只追加流水和脱敏审计。
- 新增受限 CSV/XLSX 初始库存导入：5 MiB、5000 行、固定列/工作表、dry-run、逐行报告、敏感备注拒绝、零错误整批原子执行和幂等重放。
- 新增共享库存内部端口：整批预留、释放、确认消耗、单笔过期和到期批量扫描。所有多行命令按稳定 `(warehouse_id, sku_id)` 顺序加行锁，余额不足时整批回滚。
- 新增 worker 数据库轮询执行器，逐商城读取到期预留并使用固定过期动作键安全重试；轮询间隔和批大小已进入严格环境配置。数据库轮询满足当前部署模型，未新增 BullMQ。
- 新增三语 PC 管理工作台，覆盖库存指标/预警、筛选、调整二次确认、仓库管理、流水、导入报告及加载/空/错误/冲突/重试状态；响应式布局通过管理端 Chromium 验证。
- 商品侧新 SKU 在启用默认仓创建显式零余额，不制造非零库存；库存/流水/预留/购物车已有引用时阻止物理替换 SKU，保护历史事实。

## 2. 安全与一致性证据

- 所有库存 API 先从管理员令牌、商城 Header 和 `store_id` 解析可信商城上下文，再在强制 RLS 事务中查询或写入；测试覆盖未认证拒绝、独立调整权限和跨商城空结果。
- `inventory_operations` 使用 `(store_id, operation_key)` 唯一键与 SHA-256 请求 hash；相同请求重放首次快照，相同键不同请求返回冲突，并发同键不会创建第二笔流水。
- 并发预留测试证明总预留量不超过可售量；释放、消费和过期遵循不可逆终态，重复执行不会重复恢复库存。
- 调整、导入、余额变化、流水和审计同事务；导入包含任一错误行时不产生部分余额、动作或流水。
- 备注在 API 边界拒绝明显手机号/Token 形式；审计快照不保存库存备注，密钥和令牌未写入代码、日志或测试快照。

## 4. 兼容、回滚与限制

- 本阶段复用 M3.2 已部署 schema，没有新增数据库迁移；M2 目录接口保持兼容。应用回滚时停止库存路由和过期 worker，但保留全部库存事实。
- 已产生的库存流水、动作和预留终态不可通过脚本删除或原地改写；错误库存只能以新的受审反向调整修正。
- M3.3 内部预留端口尚未绑定订单，真实下单、COD、支付与退款恢复库存由 M4 在订单事实事务中接入；本阶段没有伪造订单或销量。
- Zalo 宿主真机、生产对象存储/CDN、生产凭据、外部支付/物流和越南专业合规复核仍是上线前置条件，本报告不将 Web E2E 视为真机验收。
