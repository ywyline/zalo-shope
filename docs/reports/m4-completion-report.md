# M4 地址、结算、订单与 COD 阶段完成报告

> 状态：已按批准计划完成仓库内实现与本地自动化收口；外部上线验收有保留
>
> 日期：2026-07-23
>
> 范围：`docs/plans/m4-implementation-plan.md`；仅启用 COD，不包含真实线上支付、物流、退款、售后或生产发布。

## 1. 已交付

- 新增商城隔离的三级行政区主数据、数据库父子层级守卫和地址复合外键。买家端按省/市 -> 区/县 -> 坊/社三级联动选择；服务端忽略客户端兼容名称，只接受当前商城已启用的完整父链。未知省份也不能进入偏远地区策略，避免绕过附加费。
- 地址按商城和会员强制 RLS；收货人、规范化 `+84/+86` 手机号和详细地址使用 AES-256-GCM 密文，手机号查重使用 HMAC，接口只返回掩码手机号。地址支持默认项、软删除和乐观锁。
- Checkout 在服务端重新加载地址、行政区、商品/SKU、默认履约仓库存、促销/会员券和配送/COD 策略，以整数 VND 输出商品、优惠、运费、偏远附加费和最终应付分解。严格 DTO 拒绝客户端金额、库存、商城和状态事实。
- COD 下单使用商城/会员/操作幂等键、请求 hash、事务级 advisory lock 和 `SERIALIZABLE` 有界重试；订单、行、加密地址/价格/政策/券快照、状态转换、购物车转换、会员券核销、库存预留和幂等响应在同一事务内完成。
- 买家可查询订单列表/详情并取消待确认订单。管理员按 `store.orders.*` 查询、确认 COD、取消、关闭、维护备注/标签；敏感地址读取及高风险写入均记录审计。重复取消和确认返回既有终态，不重复释放、消费或恢复库存。
- worker 先按库存终态事实过期/释放预留，再关闭待确认订单或补偿推进已消费订单。COD 确认按领域状态机记录 `CONFIRMED -> PENDING_FULFILLMENT`，取消待履约订单追加一次 `RESTORE` 反向库存动作。
- Mini App 提供三语地址、结算、结果、订单列表/详情/取消页面；管理端提供三语订单与配送策略工作台、金额上限、标签和高风险确认。双商城继续复用同一代码并由主题和配置表达差异。

## 2. 数据、权限与契约

M4 共增加七条前向迁移：

| 迁移                                          | 作用                                               |
| --------------------------------------------- | -------------------------------------------------- |
| `20260723100000_m4_checkout_orders`           | 配送策略、加密地址、订单聚合、快照、转换和幂等记录 |
| `20260723110000_m4_order_reservation_binding` | 订单与库存预留同商城绑定                           |
| `20260723120000_m4_checkout_atomicity`        | 购物车转换、订单事实和事务完整性保护               |
| `20260723130000_m4_coupon_redemption_guard`   | 仅允许完整的一次 `CLAIMED -> USED` 会员券核销      |
| `20260723140000_m4_permission_catalog`        | 登记订单/配送四项权限，不给生产角色自动扩权        |
| `20260723150000_m4_administrative_area_guard` | 商城行政区、RLS、来源版本和父子层级触发器          |
| `20260723160000_m4_address_area_references`   | 地址省/区/坊代码到同商城主数据的复合外键           |

生产权限目录新增 `store.orders.read/manage` 与 `store.delivery.read/manage`。local/test seed 的系统商城角色显式取得这些权限；生产角色必须由授权流程单独分配。

公开契约记录于 `docs/api/openapi.m4.yaml`，数据定义见 `docs/database/m4-data-dictionary.md`，权限边界见 `docs/security/m4-permission-matrix.md`。新增行政区读取接口只接受认证会员和可信商城 Header；地址名称兼容字段已标记为弃用，服务端名称为唯一事实。

## 4. 迁移、兼容与回滚

- M4 对既有 M1-M3 契约做向后兼容新增；客户端地址名称仍可暂时提交，但被服务端忽略。应用回滚可关闭 M4 路由并保留 M3 API；已创建订单继续由兼容 worker 处理。
- 行政区导入必须先建立省级，再建立区级和坊/社级；`source_version` 非空。没有有效主数据时地址写入失败，不能用客户端代码、默认零运费或测试 fixture 降级。
- 地址、订单、快照、状态转换、券核销和库存流水属于业务事实。存在事实时 `down.sql` 以 SQLSTATE `55000` 拒绝；生产或已有事实环境只允许向前修复，库存错误使用受审反向动作。
- M4 权限迁移只登记 code。生产回滚权限目录前必须确认没有角色引用和业务事实，不能通过删除权限绕过历史审计。
