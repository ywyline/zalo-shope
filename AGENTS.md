# AGENTS.md

本文件定义 Codex 在本仓库中的长期开发规则。适用于仓库根目录及所有子目录；如未来某个子目录存在更具体的 `AGENTS.md`，则该子目录文件只覆盖其明确声明的部分。

## 1. 项目总原则

- 项目总需求是根目录的 `REQUIREMENTS.md`。开始任何产品、架构、数据或实现任务前，必须先读取与任务相关的章节。
- 本项目是一套面向越南市场的 Zalo 多品牌商城底座，分别部署美妆商城和服装商城。
- 采用一套代码、多商城独立配置的模式；不得复制两套长期分叉的业务代码。
- 越南语是默认语言，中文和英文是完整支持语言；结算货币是 VND。
- 项目是多品牌自营零售商城，不得擅自改成第三方商家入驻平台。
- 页面必须美观、移动端优先并符合 Zalo Mini App 使用习惯；不能只完成接口而交付粗糙占位页面。

## 2. 指令和需求权威顺序

发生冲突时按以下顺序处理：

1. 用户在当前任务中的明确指令。
2. 本 `AGENTS.md`。
3. `REQUIREMENTS.md`。
4. 已批准的架构、数据模型、执行计划和设计文档。
5. 现有代码、测试和历史实现。

如果用户的新要求会改变商业模式、商城隔离、核心状态机、数据模型、支付金额、库存原则、权限边界或合规范围：

- 先说明影响和需要变更的文档。
- 更新 `REQUIREMENTS.md` 或形成明确的需求变更记录。
- 在需求和计划未明确前，不擅自实施不可逆的架构变更。

普通、局部、可逆且不改变产品范围的实现细节，可以基于现有约束自行做出合理决定并记录。

## 3. 每次任务的标准流程

### 3.1 读取上下文

开始工作前：

- 阅读 `REQUIREMENTS.md` 中与任务相关的章节。
- 阅读本文件。
- 查找并阅读当前执行计划、架构文档、数据库文档和相关测试。
- 检查当前工作区状态，保留用户已有修改，不覆盖无关改动。
- 先了解现有代码和约定，再决定是否新增依赖、目录或抽象。

### 3.2 先规划的任务

以下任务必须先形成计划，再进入编码：

- 项目首次脚手架和技术栈确定。
- 多商城/多租户架构。
- 数据库整体结构或大规模迁移。
- 订单、支付、库存、物流、退款、售后状态机。
- Zalo 登录、Checkout SDK、OA、分享能力。
- 第三方支付或物流商接入。
- 权限体系、安全和个人数据处理。
- 跨多个模块或预计需要多个里程碑的功能。

计划至少说明：

- 目标和非目标。
- 涉及模块与文件。
- 数据模型和接口变化。
- 迁移、兼容和回滚方案。
- 风险、未决问题和外部依赖。
- 测试与验收方法。

如果用户明确要求“只规划”，不得修改业务代码。

### 3.3 实施

- 按已批准计划和里程碑逐步实现。
- 保持改动聚焦，不顺便重写无关模块。
- 优先使用清晰、可维护的实现；避免为了未来假设建立过度抽象。
- 复用公共商城能力，通过配置、主题和行业属性扩展美妆/服装差异。
- 新增依赖前确认必要性、维护状态、许可证和运行环境兼容性。
- 不得用静态假数据、永远成功的模拟回调或硬编码结果冒充已完成集成。
- 尚未取得真实第三方凭据时，使用明确标识的适配器和测试环境，并记录生产接入前置条件。

### 3.4 验证

完成实现后必须：

- 运行与改动相关的测试。
- 运行项目已有的 lint、格式化、类型检查和构建命令。
- 检查最终代码差异，排查遗漏、回归、调试代码和敏感信息。
- 对照 `REQUIREMENTS.md` 和任务验收条件逐项确认。
- 如有无法运行的检查，明确说明原因、影响和人工验证方式，不得宣称全部通过。

## 4. 架构边界

### 4.1 多商城隔离

- 商品、品牌、类目、SKU、库存、订单、支付、物流、营销、会员关系、页面配置和报表必须带有明确商城归属。
- 所有面向商城数据的服务端查询和写入必须校验 `store_id`/`tenant_id`，不得仅依赖前端传值或界面过滤。
- 普通商城管理员只能访问授权商城；超级管理员的跨商城能力必须显式授权并记录日志。
- 缓存键、队列消息、定时任务、文件路径、对象存储和导出任务同样必须包含商城隔离维度。
- 必须编写跨商城越权与数据泄漏测试。

### 4.2 领域模块

保持领域边界清晰，至少区分：

- 商城与主题配置。
- 身份、会员、管理员与 RBAC。
- 品牌、类目、属性、SPU 与 SKU。
- 库存与仓库。
- 购物车、促销与价格计算。
- 订单、支付、退款与财务流水。
- 物流、COD 与履约。
- 售后。
- 内容装修、国际化、搜索与分享。
- 报表、审计与通知。

不要把支付、物流和 Zalo SDK 的供应商代码散落在业务层；通过稳定的适配器接口接入，并保留供应商原始流水和回调记录。

### 4.3 数据原则

- 使用稳定的内部 ID；面向用户的订单号、SKU 编码和物流单号与内部主键分离。
- 金额使用整数 VND 或明确的定点数类型，禁止使用二进制浮点数计算订单金额。
- 订单必须保存下单时的商品、品牌、SKU、价格、优惠、地址和政策快照。
- 数据库约束、唯一索引和事务应保护关键业务不变量，不能只依靠前端校验。
- 删除已被订单或业务记录引用的品牌、类目、商品等实体时，默认采用停用或软删除。
- 所有数据库迁移应可审查、可回滚或有明确的向前修复方案。

## 5. 高风险业务规则

### 5.1 价格与促销

- 订单价格必须由服务端根据当前有效规则重新计算。
- 前端提交的价格、折扣、运费和应付金额仅作为展示信息，不能作为可信结算依据。
- 优惠券、满减、商品折扣、运费优惠必须有明确叠加、互斥、适用商城、品牌、类目和商品规则。
- 每次价格计算应可解释，并能够还原订单价格组成。

### 5.2 库存

- 区分实际库存、锁定库存和可售库存。
- 下单锁库存、超时释放、支付或 COD 确认后扣减的时机必须由明确状态机控制。
- 支付重复回调、任务重试和并发请求不能导致重复扣减或超卖。
- 取消、退款、拒收和退回是否恢复库存必须依据订单与商品状态判断。

### 5.3 支付与退款

- 服务端创建支付单、验证签名、核对金额、商城归属和订单状态。
- 支付回调必须幂等；不能仅凭客户端跳转结果标记支付成功。
- 回调可能丢失，必须支持主动查询和对账补偿。
- 支付流水、业务订单和退款流水分离建模，保留原始供应商响应。
- 密钥和商户凭据不得进入前端、日志、测试快照或版本库。

### 5.4 COD、物流与售后

- COD 支持金额、商品、地区、商城和用户风险限制。
- 记录拒收、退回费用、COD 应收、回款和对账差异。
- 物流状态必须映射到内部统一状态，不直接让供应商状态污染订单领域。
- 退款、退货、换货必须有独立售后状态机和审计记录。

## 6. Zalo 和第三方集成

- 只使用 Zalo 官方允许的登录、权限、Checkout SDK、OA 和分享能力。
- 权限按实际场景申请；拒绝授权时提供合理的手动或降级流程。
- 分享由用户主动触发，不实现强制、自动或奖励诱导分享。
- 外部链接、支付方式和通知能力必须符合 Zalo Mini App 审核政策。
- 接口签名、时间戳、重放保护、回调重试、超时和限流必须有测试。
- 测试环境与生产环境配置完全分离。

## 7. 国际化与越南本地化

- 禁止在业务组件中散落硬编码用户文案；使用统一的 i18n key。
- 越南语是默认和发布必备语言；中文、英文缺失时按需求回退越南语。
- 数据库、搜索、排序、导入导出和通知必须正确支持越南语变音符号。
- 商品、品牌、类目、活动、协议、通知和分享内容均考虑三语结构。
- 地址模型支持越南省/市、区/县、坊/社和详细地址。
- 日期、手机号、VND 价格与数量格式按越南本地习惯显示。

## 8. UI/UX 与可访问性

- 延续已确认的视觉方向：美妆商城精致柔和，服装商城简洁时尚；公共组件通过主题变量复用。
- 优先使用真实的响应式布局和设计系统组件，不用大量一次性内联样式拼页面。
- 买家端以移动端单手操作为基准；重要操作需有加载、成功、失败、空状态和重试反馈。
- SKU、筛选和结算流程应减少跳转和输入负担。
- 图片需要合理尺寸、懒加载、失败占位和可替代文本。
- 核心流程需要基本键盘操作、可读焦点和足够的颜色对比度。
- 不得为了视觉效果牺牲价格、促销、库存和政策信息的清晰度。

## 9. 测试要求

根据改动风险选择测试层级，关键业务不能只依赖手工测试。

至少覆盖：

- 单元测试：价格、促销、状态转换、权限判断、地址和本地化格式。
- 集成测试：数据库约束、库存锁定、支付/退款幂等、物流状态映射、商城隔离。
- API 测试：身份认证、输入校验、越权、重复请求、错误响应。
- E2E 测试：首页到下单、线上支付测试流程、COD、取消、发货、收货和售后。
- 安全测试：跨商城访问、RBAC、敏感字段、签名验证、重放和金额篡改。
- UI 测试：三种语言、常见 Android/iPhone 尺寸、加载/空/错误状态。

修复缺陷时优先先添加能够复现问题的回归测试，再实施最小安全修复。

## 10. 安全、隐私与合规

- 采用最小权限原则，管理端敏感操作需要审计；高风险操作按设计增加二次确认。
- 用户手机号、地址、身份标识和支付相关数据按敏感数据处理，日志默认脱敏。
- 不收集实现功能不需要的数据；保留用户同意、撤回、更正、删除和注销路径。
- 依赖和输入文件均视为不可信；验证类型、大小、内容和访问权限。
- 美妆、美瞳、口腔和私密护理商品的合规字段与上架门禁不得被普通运营绕过。
- 合规逻辑需要配置化并保留审核记录，但不得把本需求文档当作正式法律意见。

## 11. 项目命令

项目采用 Node.js 24 LTS 与 Corepack 管理的 pnpm workspace。以下命令必须从仓库根目录运行：

- 包管理器版本：`corepack pnpm --version`。
- 安装依赖：`corepack pnpm install --frozen-lockfile`。
- 启动本地依赖：`corepack pnpm infra:up`；停止：`corepack pnpm infra:down`。
- 本地开发：`corepack pnpm dev`。
- 单元测试：`corepack pnpm test:unit`。
- 基础设施集成测试：先运行 `corepack pnpm infra:up`，再运行 `corepack pnpm test:integration`。
- 浏览器 E2E：首次运行 `corepack pnpm test:e2e:install` 安装 Chromium/WebKit；启动基础设施后运行 `corepack pnpm test:e2e`。
- lint：`corepack pnpm lint`。
- 格式检查：`corepack pnpm format:check`；格式化：`corepack pnpm format`。
- 类型检查：`corepack pnpm typecheck`。
- 生产构建：`corepack pnpm build`。
- 全部静态、单元与构建验证：`corepack pnpm verify`。
- 数据库客户端生成：`corepack pnpm db:generate`；schema 校验：`corepack pnpm db:validate`。
- 开发迁移：`corepack pnpm --filter @zalo-shop/database migrate:dev`。
- 部署迁移：`corepack pnpm --filter @zalo-shop/database migrate:deploy`。
- 本地/测试种子：`NODE_ENV=test corepack pnpm --filter @zalo-shop/database seed`（PowerShell 使用 `$env:NODE_ENV='test'`）。
- 创建首个管理员：配置一次性 `ADMIN_EMAIL`、`ADMIN_PASSWORD`、`ADMIN_TOTP_SECRET` 后运行 `corepack pnpm --filter @zalo-shop/database admin:create`；禁止将这些值写入仓库或 shell 历史。
- 生产依赖审计：`corepack pnpm audit --prod --audit-level high`。

M1 已提供身份/RBAC 数据迁移、种子及 API 集成测试。迁移回滚只允许在无真实业务数据的 local/test 环境按迁移目录中的 `down.sql` 人工执行；产生身份、同意或审计数据后只允许向前修复。M2 已提供基于 Chromium/WebKit 的可重复浏览器 E2E；它覆盖 Web 预览但不能替代 Zalo 宿主真机验收。项目直接使用 `corepack pnpm`，不依赖需要管理员权限的全局 pnpm shim；Windows 也可使用 `corepack.cmd pnpm`，详见 `README.md`。

## 12. 文档与变更管理

- 架构确定后维护架构文档、领域边界和关键决策记录。
- 数据库变化同步更新 schema/ERD、迁移和字段说明。
- API 变化同步更新接口契约和调用方。
- 新增环境变量同步更新示例配置，示例中不得包含真实密钥。
- 已批准计划发生变化时更新计划状态和原因。
- 不因实现方便而静默改变 `REQUIREMENTS.md` 中的业务规则。

## 13. Git 和代码审查

- 不覆盖或回退用户未授权的现有修改。
- 提交保持小而聚焦，提交信息说明业务意图。
- 不提交密钥、生产数据、构建产物、临时文件、调试日志或无关格式化改动。
- 完成前检查 diff，重点关注跨商城泄漏、权限绕过、金额错误、重复扣库存、状态跳跃和错误信息泄露。
- 如用户要求发布或创建 PR，再执行提交、推送和 PR 流程；普通开发任务不自动发布外部变更。

## 14. 完成定义

只有同时满足以下条件，任务才可以标记完成：

- 实现与当前任务、`REQUIREMENTS.md` 和已批准计划一致。
- 相关测试已新增或更新并通过。
- lint、类型检查和构建等适用检查已通过。
- 多商城隔离、RBAC、金额、库存和状态机等受影响风险已验证。
- 三语与移动端体验在受影响页面得到验证。
- 无调试代码、虚假成功路径、明文密钥和未说明的临时方案。
- 文档、配置、迁移和接口契约已同步。
- 最终汇报列出改动、验证证据、已知限制和后续事项。

“代码已生成”不等于“任务已完成”。无法验证的部分必须明确标注为未验证。

## 15. Superpowers 或其他工作流插件

- 本文件定义的流程不依赖 Superpowers，也足以作为本项目的基础严谨工作流。
- 如果会话中安装并明确要求使用 Superpowers，可使用其规划、TDD、系统化调试和代码审查技能。
- 插件工作流不得覆盖 `REQUIREMENTS.md`、本文件、已批准计划或用户当前指令。
- 不因插件要求自动扩大任务范围、创建无关分支或执行外部发布。
